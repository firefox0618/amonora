# Целевая структура `site` и `client`

## Назначение

Этот документ фиксирует безопасную целевую файловую структуру для будущего разделения `landing` на два отдельных приложения:

- `site`
- `client`

На этом шаге логика ещё не переносится.

## Созданный каркас

В репозитории создан новый верхнеуровневый каталог:

- `apps/`

Внутри него созданы:

- `apps/site/`
- `apps/client/`
- `apps/shared/`

## Зачем нужен `apps/`

Каталог `apps/` нужен как целевая точка для самостоятельных приложений проекта.

Это позволяет дальше переносить контуры по одному, не ломая текущий runtime и не смешивая новые границы с текущим монолитом.

## Будущая структура `apps/site`

Ниже зафиксирован целевой вид, к которому нужно двигаться поэтапно.

```text
apps/site/
├── README.md
├── app.py
├── templates/
│   ├── index.html
│   └── legal.html
└── static/
    ├── landing.css
    ├── landing.js
    ├── manual/
    └── ...
```

### Будущий entrypoint

- `apps/site/app.py`

### Что должен обслуживать

- `/`
- `HEAD /`
- `/legal/{page_key}`
- `/manual`
- `/{filename}.txt`

## Будущая структура `apps/client`

```text
apps/client/
├── README.md
├── app.py
├── templates/
│   ├── client_subscription_shell.html
│   └── client_happ_wrapper.html
└── static/
    └── client-app/
```

### Будущий entrypoint

- `apps/client/app.py`

### Что должен обслуживать

- `/api/public/subscriptions/{token}/summary`
- `/api/public/subscriptions/{token}/touch`
- `/sub/{token}`
- `/happ/add`
- `/{token}`

## Временные общие части

На этапе разделения часть кода может временно понадобиться обоим приложениям.

Для этого зафиксирован переходный каталог:

- `apps/shared/`

Туда при необходимости можно переносить только реально общие части, чтобы не копировать код между `site` и `client`.

### Что допустимо временно держать в `apps/shared`

- общий bootstrap публичных FastAPI-приложений;
- общая настройка шаблонов;
- общие middleware публичных поверхностей;
- общие helper-функции для URL, request-id и обработки ошибок;
- общие функции рендеринга markdown-страниц, если они понадобятся обеим сторонам.

### Что нельзя превращать в `apps/shared`

- VPN provisioning;
- бизнес-логику подписок;
- платёжные webhooks;
- bridge access;
- тяжёлую доменную логику, которой не место в публичных приложениях.

## Что важно на этом этапе

Создание каркаса `apps/` не меняет текущий runtime.

Пока продолжают работать текущие входные точки:

- `landing/main.py`
- существующие шаблоны и статика в `landing/`

Новый каркас нужен для следующего шага, когда начнётся фактический перенос файлов и entrypoint'ов.

## Итог шага

На этом шаге:

- создан новый верхнеуровневый каркас `apps/`;
- зафиксированы будущие каталоги `site`, `client` и переходный `shared`;
- зафиксированы будущие entrypoint'ы;
- зафиксировано, какие части можно временно считать общими, а какие нельзя туда уносить.

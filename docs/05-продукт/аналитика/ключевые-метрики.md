# Ключевые метрики

## Обзор

Система аналитики Amonora собирает метрики через события (`AnalyticsEvent`) и агрегирует их в rollup-таблицы.

> **Примечание:** Grafana была удалена. Метрики теперь доступны через dashboard API и напрямую из БД. Rollup-таблицы (`analytics_daily_*`) продолжают заполняться через `analytics_refresh.py`.

**Источник:** `backend/core/analytics.py`, `ops/analytics_refresh.py`, `dashboard/analytics.py`

## Список KPI

### MAU (Monthly Active Users) — Месячные активные пользователи

**Описание:** Количество пользователей с активной подпиской за месяц.

**Где считается:** `analytics_runtime_status` → `growth_active_users`

**Формула:** Подсчёт пользователей со статусом `paid_active` или `trial_active` на текущий момент.

**Источник:** `ops/grafana/build_suite.py` — панель «Активные подписки»

### LTV (Lifetime Value) — Пожизненная ценность

**Описание:** Средний доход от одного платящего пользователя за всё время.

**Формула:** `общая выручка / количество уникальных платящих пользователей`

**Где смотреть:** `dashboard/analytics.py` — `revenue_per_user`

**Обновление:** При каждом `analytics_refresh`

### Churn Rate (Отток)

**Описание:** Процент пользователей, потерявших подписку за период.

**Где смотреть:** Grafana дашборд «Удержание и отток» (`amonora-retention-churn`)

**Формула (косвенная):** Через когортный анализ — `AnalyticsCohortRetention` таблица отслеживает удержание по дням (0, 1, 3, 7, 14, 30, 60, 90 дней).

**Когорты:**
- `trial_started` — пользователи, начавшие trial
- `subscription_activated` — пользователи, активировавшие платную подписку

**Источник:** `backend/core/analytics.py` — `COHORT_PERIOD_DAYS`, `COHORT_TYPES`

### Конверсия Trial → Paid

**Описание:** Процент trial-пользователей, перешедших в платную подписку.

**Формула:**
```
trial_to_paid_percent = (пользователи с paid_active И trial_used) / (все пользователи с trial_used) * 100
```

**Где смотреть:**
- Dashboard analytics → `users.trial_to_paid_percent`
- Grafana дашборд «Главная Amonora» → «Конверсия в оплату, %»

**Источник:** `dashboard/analytics.py` — `_calculate_user_metrics()`

### ARPU (Average Revenue Per User) — Средний доход на пользователя

**Описание:** Средний доход на одного пользователя за период.

**Формула:** `общая выручка / общее количество пользователей`

**Где смотреть:** `dashboard/analytics.py` — `revenue_per_user` (доход на платящего пользователя)

### MRR (Monthly Recurring Revenue) — Месячный регулярный доход

**Описание:** Сумма подтверждённых платежей за период.

**Где смотреть:**
- Dashboard analytics → `payments.revenue`
- Grafana дашборд «Выручка и монетизация» (`amonora-revenue-monetization`)

**Формула:** `SUM(amount) для confirmed платежей за период`

**Разбивка:**
- По тарифам (`tariff_revenue`)
- По типу платежа (новый vs продление)
- По методу оплаты

**Источник:** `dashboard/analytics.py` — `_calculate_payment_metrics()`

## Как обновляются метрики

### Автоматическое обновление

Скрипт `ops/analytics_refresh.py` запускает `run_analytics_maintenance()`:

```bash
python -m ops.analytics_refresh          # Обычное обновление
python -m ops.analytics_refresh --backfill     # С заполнением истории
python -m ops.analytics_refresh --full-refresh # Полная перестройка
```

**Режимы:**
- `--backfill` — заполняет исторические данные атрибуции/событий
- `--full-refresh` — перестраивает rollup-таблицы из полного лога событий

### Real-time триггер

При каждом событии (`emit_analytics_event`) вызывается `_request_near_realtime_refresh_for_event()`, который запускает быстрое обновление.

**Источник:** `backend/core/analytics.py` — `_request_near_realtime_refresh_for_event()`

### Периодичность

Grafana дашборды обновляются каждые **5 минут** (`refresh: "5m"`).

Rollup-таблицы:
- `analytics_daily_stage_counts` — ежедневные срезы по этапам
- `analytics_daily_revenue` — ежедневная выручка
- `analytics_daily_revenue_segment` — сегментированная выручка
- `analytics_daily_connection` — подключения
- `analytics_daily_payment_failure_reason` — причины отказов
- `analytics_cohort_retention` — когортное удержание

**Источник:** `backend/core/models.py` — модели аналитики

## Целевые значения

Целевые значения **не зафиксированы** в коде проекта. Требуется уточнение у команды.

Ориентиры из структуры дашбордов:
- Конверсия в оплату — отображается как процент, пороговые значения настраиваются в Grafana
- Целостность атрибуции — отображается в процентах, цель близка к 100%
- Разрыв оплата → подключение — чем меньше, тем лучше

**Требуется уточнение:** формальные целевые KPI не определены в коде.

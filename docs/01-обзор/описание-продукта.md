# Описание продукта Amonora

## Что такое Amonora

Amonora — экосистема цифровых сервисов. Текущий основной продукт — **Telegram-first VPN-сервис** с **единой подпиской**, где весь путь пользователя построен вокруг Telegram-бота `@amonora_bot`.

Клиент получает trial, покупает подписку и получает **единую ссылку** (`client.amonoraconnect.com/<token>`), которая автоматически включает все доступные серверы (Германия, Дания, Эстония) с failover-маршрутизацией. Выбор страны, протокола и режима — **автоматический**, клиент не управляет этими параметрами вручную.

Публичный сайт **amonoraconnect.com** служит витриной продукта, точкой входа для bridge-access (бесплатный ключ на 24 часа, если Telegram недоступен) и хостингом tokenized subscription page.

Проект развивается как **modular monolith** — одна связанная система без разделения на микросервисы.

## Ключевой принцип: единая подписка

Подписка Amonora — это **не выбор одного сервера**, а **доступ ко всей инфраструктуре сразу**:

- **Один токен** → все серверы (DE, DK, EE) включены в subscription feed
- **Happ-клиент** автоматически получает список всех серверов и выбирает оптимальный
- **Ручной импорт** — можно скопировать individual connection URI, но маршрутизацией управляет система
- **Failover** — если один сервер недоступен, трафик перенаправляется на другие
- **Клиент не выбирает** страну, протокол (VLESS/Trojan) или режим (Стабильный/Мобильный/Резерв) — это внутренняя логика системы

## Основной сценарий (клиент)

### Онбординг нового пользователя

1. **Вход.** Пользователь приходит через сайт `amonoraconnect.com` или напрямую в `@amonora_bot`
2. **Регистрация.** Нажимает `/start` → бот показывает пользовательское соглашение → пользователь принимает
3. **Trial intro.** Бот показывает экран пробного периода: «Бесплатный пробный период на 3 дня!»
4. **Подписка на канал.** Пользователь подписывается на канал `@amonora_new` → нажимает «Проверить подписку»
5. **Активация триала.** Бот активирует trial на 3 дня → показывает экран готовности с кнопками «Ключ», «Инструкция», «Поддержка»

### Получение единой ссылки

6. **Главное меню.** Пользователь видит: «Моя подписка», «Ключ», «Продлить», «Информация», «Поддержка», «Бонусная система»
7. **Экран «Ключ».** Пользователь нажимает «Ключ» → видит:
   - **[Открыть подписку]** — URL `https://client.amonoraconnect.com/<token>` (браузер)
   - **[Открыть в Happ]** — URL `https://client.amonoraconnect.com/happ/add?sub=...` (deep link)
   - **[Скопировать ссылку]** — копирует connection URI для ручного импорта
   - **[Мои устройства]** — список привязанных устройств

8. **Подключение.** Клиент открывает ссылку в браузере → видит страницу с:
   - Статусом подписки и сроком действия
   - Инструкции по установке Happ для всех платформ (Android, iOS, Windows, macOS, Linux, Apple TV, Android TV)
   - QR-кодом для быстрого подключения

9. **Happ-клиент.** При открытии в Happ → автоматическое добавление подписки со всеми серверами (DE, DK, EE)

### Покупка подписки

10. **Продление.** Пользователь нажимает «Продлить» → выбирает тариф (1м / 3м / 6м / 12м)
11. **Оплата.** Выбирает способ: СБП (авто через Platega), СБП (ручная), Криптовалюта, Баланс (RUB), Telegram Stars
12. **Активация.** После подтверждения платежа → подписка активируется, VPN-доступ обновляется

### Поддержка

13. **Обращение.** При проблемах → кнопка «Поддержка» → переход в `@amonora_support_bot`
14. **Тикет.** Создаётся тикет с номером → оператор отвечает → диалог ведётся до закрытия

## Архитектура интерфейса бота

Основной роутер (`bot/router.py`, ~4900 строк) содержит весь UI-код клиентского бота:

- **Главное меню** — inline-клавиатура с 6 кнопками (подписка, ключ, продлить, информация, поддержка, бонусы)
- **Система экранов** — каждое состояние UI = фото + текст + клавиатура, идентифицируется `screen_key`
- **Единая подписка** — токен на пользователя, генерирует URL для браузера и Happ
- **Onboarding** — соглашение → триал → готовность
- **Платежи** — тариф → способ оплаты → подтверждение

### Legacy-слой (из `bot/handlers/`)

Старые обработчики (`start.py`, `devices.py`, `protocol.py`, `tariffs.py`) **недостижимы** для обычных пользователей, так как основной роутер подключён первым. Этот слой сохраняется как shared-логика для dashboard и внутренних вызовов.

## Как устроен проект (структура файлов)

```
amonora/
├── backend/              # Общее ядро: модели, схема БД, доменные сущности
│   └── core/             # models.py, schema.py, database.py, analytics.py, ...
├── bot/                  # Клиентский бот @amonora_bot
│   ├── router.py         # Основной UI-роутер (~4900 строк)
│   ├── handlers/         # Legacy-обработчики (shared-логика)
│   ├── keyboards/        # Inline / reply клавиатуры
│   ├── middlewares/      # Middleware (логирование активности)
│   ├── utils/            # VLESS, тарифы, тексты, регионы, QR, рефералы, ...
│   ├── public_subscription.py  # Логика unified subscription
│   ├── payment_flow.py   # Общие платежные потоки
│   ├── config.py         # Конфигурация из ENV
│   └── main.py           # Точка входа (router.py подключён первым!)
├── support_bot/          # Бот поддержки @amonora_support_bot
├── control_bot/          # Контрольный бот @amonora_control_bot
├── test_bot/             # Тестовый бот (независимый от основного)
│   ├── main.py           # Точка входа
│   ├── router.py         # Свой роутер (тестовые профили)
│   ├── profiles.py       # Генерация тестовых конфигов
│   └── access.py         # Проверка доступа (allowlist)
├── dashboard/            # Backend админки (FastAPI)
├── dashboard/ui/         # Frontend админки (Next.js 16)
├── client_ui/            # Исходники tokenized client page (React + Mantine)
├── landing/              # Публичный сайт-витрина + webhook'и
│   └── static/client-app/ # Скомпилированная клиентская страница (SPA)
├── ops/                  # Операционные конфиги (systemd, nginx, env, backup)
├── docs/                 # Документация для людей
```

## Зависимости между компонентами

```
bot (v2_router) → backend models → PostgreSQL
support_bot → backend models → PostgreSQL
control_bot → backend models → PostgreSQL
dashboard → backend models + PostgreSQL
dashboard/ui → dashboard API → PostgreSQL
landing → backend models + PostgreSQL
client_ui → (статика через landing)
ops → systemd / nginx / runtime
```

## Источник истины

**PostgreSQL (`amonora_db`)** — единственный source of truth. `dashboard/ui`, `landing`, и `client_ui` не хранят свою бизнес-логику и не являются источником истины.

VPN-ноды вынесены на отдельные серверы. На них **не** живут: боты, dashboard, PostgreSQL.

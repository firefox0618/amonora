# Классификация причин ремонта VPN

Причины, по которым устройство помечается как «требует ремонта» или ремонт не удаётся.

## Основные причины (ремонт + отображение в панели)

| Причина | Когда возникает | Действие |
|---------|----------------|----------|
| `post_payment_sync_failed` | После оплаты не удалось синхронизировать клиент на ноде | Ручной re-sync |
| `post_payment_access_incomplete` | Оплата прошла, но клиент не получил доступ | Проверить provisioning |
| `manual_repair_sync_failed` | Ручной ремонт: sync не прошёл | Проверить ноду |
| `manual_repair_no_access` | Ручной ремонт: доступа нет после sync | Проверить подписку |
| `manual_repair_no_devices` | У пользователя нет устройств для ремонта | Создать устройство |

## Внутренние причины (только логи, не показываются)

| Причина | Описание |
|---------|----------|
| `auto_repair_success` | Автоматический ремонт прошёл успешно |
| `auto_repair_failed` | Автоматический ремонт не удался после N попыток |

## Нормализация

Причины нормализуются при записи. Legacy-aliases конвертируются на чтение:
- `sync_failed` → `post_payment_sync_failed`
- `access_incomplete` → `post_payment_access_incomplete`
- `repair_no_devices` → `manual_repair_no_devices`

## Хранение

| Таблица | Описание |
|---------|----------|
| `vpn_repair_events` | События ремонта (user_id, result, reason) |
| `device_compensation_jobs` | Асинхронные задачи компенсации |

## Автоматический ремонт

`bot/device_compensation.py` обрабатывает компенсационные задачи:

| Действие | Описание |
|----------|----------|
| `cleanup_created_device` | Удалить частично созданное устройство при сбое |
| `finalize_created_device` | Завершить создание если provisioning частично успешен |
| `restore_deleted_device` | Восстановить удалённое устройство при сбое |

Параметры:
- Max попыток: 10
- Backoff: `attempt × 5 минут` (макс. 60 минут)
- Stale lock: 900 секунд

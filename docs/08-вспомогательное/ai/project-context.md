# Контекст проекта

## Обзор проекта
Amonora — экосистема цифровых сервисов.
Текущий продукт: VPN-сервис с основным клиентским путём через Telegram-бота `@amonora_bot`.

## Основные компоненты системы

| Компонент | Расположение | Назначение |
|-----------|-------------|------------|
| Основной бот | `bot/` | Точка входа для пользователей |
| Support-бот | `support_bot/` | Обращения и тикеты поддержки |
| Control-бот | `control_bot/` | Внутренние уведомления + админ-операции |
| Публичный сайт | `landing/` | Витрина продукта, bridge-access |
| Панель управления | `dashboard/` + `dashboard/ui` | Административный backend/API + frontend |
| Бэкенд | `backend/` | Общая доменная логика, модели, ядро |
| База данных | PostgreSQL | Единый источник достоверных данных |
| Операционные скрипты | `ops/` | Развёртывание, runtime, инфраструктура |

## Архитектурная реальность

Код не полностью нормализован. Доменная логика распределена между:

- `backend/`
- `bot/`
- `dashboard/`
- `support_bot/`

Это ожидаемое состояние, которое должно учитываться при внесении изменений.

## Ключевые переходные области

- `dashboard/` и `dashboard/ui/` сосуществуют — старая и новая панели управления
- Папка `docs/` больше не используется — все документы находятся в `documentation/`

## Приоритетные цели

1. **Сохранять** рабочую функциональность
2. **Улучшать** ясность структуры
3. **Снижать** архитектурную неоднозначность
4. **Делать** изменения безопасными для людей и AI
5. **Работать** маленькими задачами, а не широкими рефакторингами

## Правила работы

- Не предполагать полную центраализацию кодовой базы
- Не рассматривать `dashboard/` как полностью унаследованный код
- Не удалять старые UI-потоки из `dashboard/` без подтверждения покрытия в `dashboard/ui/`
- Не перемещать файлы без привязки к задаче
- Не изменять runtime-пути, порты, имена сервисов без обновления `RUNBOOK.md`
- Не вносить секретные значения в репозиторий — использовать шаблоны `PRIVATE_ACCESS.template.md`
- Предпочитать небольшие, ревьюируемые, обратимые изменения

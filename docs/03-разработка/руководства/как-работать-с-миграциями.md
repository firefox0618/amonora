# Как работать с миграциями Amonora

## Обзор

Amonora использует **программные миграции** через SQLAlchemy DDL, а не Alembic. Миграции определены в `backend/core/schema.py` и применяются автоматически при старте.

---

## Где лежат миграции

Все миграции находятся в **одном файле**:

```
backend/core/schema.py
```

Структура:

```python
@dataclass(frozen=True)
class SchemaMigrationStep:
    key: str            # Уникальный идентификатор шага
    statement: str      # SQL-запрос (ALTER TABLE, CREATE INDEX, etc.)
    verify_query: str | None = None  # Запрос для проверки успешности
```

Регистри миграций:
```python
CORE_SCHEMA_MIGRATIONS = (
    SchemaMigrationStep("core_001_users_subscription_started_at", "ALTER TABLE users ADD COLUMN IF NOT EXISTS ..."),
    SchemaMigrationStep("core_002_users_subscription_expires_at", "ALTER TABLE users ADD COLUMN IF NOT EXISTS ..."),
    # ... 100+ шагов
)
```

Текущее количество шагов: **100+** (до `core_100+`).

Таблица отслеживания миграций:
```python
SCHEMA_MIGRATION_REGISTRY_TABLE = "schema_migration_steps"
```

---

## Как применяется схема

Функция `ensure_schema()` в `backend/core/schema.py`:

1. Проверяет что `AUTO_APPLY_SCHEMA=1` в `.env`
2. Подключается к PostgreSQL
3. Создаёт таблицу `schema_migration_steps` (если нет)
4. Читает уже применённые шаги из `schema_migration_steps`
5. Применяет каждый неприменённый шаг:
   - Выполняет `statement`
   - Если есть `verify_query` — проверяет результат
   - Записывает ключ шага в `schema_migration_steps`
6. Проверяет обязательные колонки и индексы через `REQUIRED_SCHEMA_COLUMNS` и `REQUIRED_SCHEMA_INDEXES`

Вызывается при старте всех компонентов:
```python
# bot/main.py
await ensure_schema()

# support_bot/main.py
await ensure_schema()

# control_bot/main.py
await ensure_schema()

# test_bot/main.py
await ensure_schema()
```

## Как создать новую миграцию

### Шаг 1: Определи модель

Если новая модель — добавь её в `backend/core/models.py`:

```python
class NewModel(Base):
    __tablename__ = "new_models"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(255), nullable=False)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)
```

### Шаг 2: Добавь шаг миграции

В `backend/core/schema.py`:

1. Найди последний шаг в `CORE_SCHEMA_MIGRATIONS`
2. Добавь новый шаг:

```python
CORE_SCHEMA_MIGRATIONS = (
    # ... существующие шаги ...

    SchemaMigrationStep(
        "core_XXX_new_models_table",
        """
        CREATE TABLE IF NOT EXISTS new_models (
            id SERIAL PRIMARY KEY,
            name VARCHAR(255) NOT NULL,
            created_at TIMESTAMP NOT NULL DEFAULT NOW()
        )
        """,
        # verify_query — опционально
        """
        SELECT EXISTS (
            SELECT 1 FROM information_schema.tables
            WHERE table_schema = current_schema()
              AND table_name = 'new_models'
        )
        """,
    ),
)
```

### Шаг 3: Добавь колонку в REQUIRED_SCHEMA_COLUMNS

Если новая колонка в существующей таблице:

```python
REQUIRED_SCHEMA_COLUMNS = {
    # ... существующие ...
    "new_models": {"id", "name", "created_at"},
}
```

### Шаг 4: Добавь индекс в REQUIRED_SCHEMA_INDEXES

```python
REQUIRED_SCHEMA_INDEXES = {
    # ... существующие ...
    "idx_new_models_name",
}
```

И добавь шаг создания индекса:

```python
SchemaMigrationStep(
    "core_XXX_new_models_name_index",
    "CREATE INDEX IF NOT EXISTS idx_new_models_name ON new_models (name)",
    """
    SELECT EXISTS (
        SELECT 1 FROM pg_indexes
        WHERE schemaname = current_schema()
          AND indexname = 'idx_new_models_name'
    )
    """,
),
```

## Как накатить на dev

### Автоматически (рекомендуется)

1. Убедись что `AUTO_APPLY_SCHEMA=1` в `.env`
2. Запусти бота или dashboard:
   ```bash
   python -m bot.main
   ```
3. Схема применится автоматически при старте.

### Вручную

```python
python -c "
from backend.core.schema import ensure_schema
import asyncio
asyncio.run(ensure_schema())
print('Schema applied successfully')
"
```

### Проверка статуса миграций

```python
python -c "
import asyncio
from backend.core.database import engine
from sqlalchemy import text

async def check():
    async with engine.connect() as conn:
        # Проверка таблицы миграций
        result = await conn.execute(text(
            'SELECT key FROM schema_migration_steps ORDER BY key DESC LIMIT 10'
        ))
        for row in result:
            print(row[0])

asyncio.run(check())
"
```

## Как накатить на production

### Подготовка

1. **Сделай бэкап базы:**
   ```bash
   pg_dump -U amonora amonora_db > /backup/amonora_db_$(date +%Y%m%d).sql
   ```

2. **Проверь миграции локально:**
   ```bash
   python -c "
   from backend.core.schema import ensure_schema
   import asyncio
   asyncio.run(ensure_schema())
   "
   ```

3. **Просмотри SQL-запросы** нового шага миграции — убедись что они безопасны.

### Применение

1. **Deploy кода** на production сервер
   ```bash
   git pull
   pip install -r requirements.txt
   ```

2. **Перезапусти сервисы:**
   ```bash
   sudo systemctl restart amonora-control-bot
   # Остальные сервисы подхватят схему при старте
   ```

3. **Проверь логи:**
   ```bash
   journalctl -u amonora-control-bot -n 50 --no-pager
   ```

   Ищи строки про схему:
   - `Schema migration step applied: core_XXX_...`
   - `Schema ready`

### Если AUTO_APPLY_SCHEMA=0

В production может быть отключено автоматическое применение схемы. В этом случае:

```python
# Запусти вручную на production сервере:
cd /opt/amonora_bot
source venv/bin/activate
python -c "
from backend.core.schema import ensure_schema
import asyncio
asyncio.run(ensure_schema())
"
```

## Как откатить миграцию

### Важное предупреждение

> **Amonora НЕ поддерживает автоматический откат миграций.** Все операции используют `IF NOT EXISTS` и `ADD COLUMN IF NOT EXISTS`, что делает их идемпотентными.

### Откат добавления колонки

```sql
ALTER TABLE users DROP COLUMN IF EXISTS new_column_name;
```

### Откат создания таблицы

```sql
DROP TABLE IF EXISTS new_models CASCADE;
```

### Откат индекса

```sql
DROP INDEX IF EXISTS idx_new_models_name;
```

### Удаление записи из registry

```sql
DELETE FROM schema_migration_steps WHERE key = 'core_XXX_new_models_table';
```

### После ручного отката

Удали соответствующий `SchemaMigrationStep` из `CORE_SCHEMA_MIGRATIONS` в `backend/core/schema.py`, чтобы он не применился снова.

## Частые проблемы и решения

### Проблема: `column already exists`

**Причина:** Шаг миграции не использует `IF NOT EXISTS`.

**Решение:** Убедись что все `ALTER TABLE ADD COLUMN` содержат `IF NOT EXISTS`:
```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS new_column ...
```

### Проблема: `relation already exists`

**Причина:** Таблица уже создана, но шаг миграции не содержит `IF NOT EXISTS`.

**Решение:** Используй `CREATE TABLE IF NOT EXISTS`.

### Проблема: миграция не применяется

**Причина 1:** `AUTO_APPLY_SCHEMA=0` в `.env`.

**Решение:**
```bash
grep AUTO_APPLY_SCHEMA .env
# Если 0 — измени на 1
```

**Причина 2:** Нет прав на DDL-операции.

**Решение:** Убедись что пользователь БД имеет права `CREATE`, `ALTER`:
```sql
GRANT ALL ON DATABASE amonora_db TO amonora;
```

**Причина 3:** Verify query не проходит.

**Решение:** Проверь verify_query вручную:
```sql
-- Скопируй verify_query из SchemaMigrationStep и выполни
SELECT EXISTS (
    SELECT 1 FROM pg_indexes
    WHERE schemaname = current_schema()
      AND indexname = 'idx_users_ref_code'
);
```

### Проблема: рассинхрон schema_migration_steps и реальной схемы

**Причина:** Ручные изменения БД в обход миграций.

**Решение:**
1. Очисти registry:
   ```sql
   DELETE FROM schema_migration_steps;
   ```
2. Перезапусти сервис — все шаги с `IF NOT EXISTS` безопасно применятся снова.

### Проблема: missing required columns

**Причина:** Новая колонка определена в модели, но шаг миграции не добавлен.

**Решение:**
1. Добавь `SchemaMigrationStep` в `CORE_SCHEMA_MIGRATIONS`
2. Добавь колонку в `REQUIRED_SCHEMA_COLUMNS`
3. Перезапусти сервис

### Проблема: missing required indexes

**Причина:** Аналогично колонкам — индекс определён в REQUIRED_SCHEMA_INDEXES, но шаг миграции не создаёт его.

**Решение:** Добавь шаг создания индекса.

## Порядок миграций

Шаги выполняются **последовательно** в порядке кортежа `CORE_SCHEMA_MIGRATIONS`.

Номенклатура ключей:
```
core_001_...    — первые миграции
core_021a_...   — побочные миграции (a, b, c)
core_051_...    — миграции индексов
core_100+_...   — последние миграции
```

Каждый ключ должен быть **уникальным**. Повторный запуск пропускает уже применённые шаги.

---

## Идемпотентность

Все миграции спроектированы идемпотентно:

| Операция | Идемпотентная форма |
|----------|-------------------|
| Добавить колонку | `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...` |
| Создать таблицу | `CREATE TABLE IF NOT EXISTS ...` |
| Создать индекс | `CREATE INDEX IF NOT EXISTS ...` |
| Добавить ограничение | `DO $$ ... IF NOT EXISTS ...` |

Это означает что `ensure_schema()` можно безопасно вызывать **многократно** — она не сломает существующую схему.

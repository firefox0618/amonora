# REPO RULES

## Зачем нужен этот документ

Этот документ фиксирует правила чтения репозитория.

Его задача — не описать весь проект заново, а убрать путаницу:

- что в репозитории считается архитектурой;
- что не считается архитектурой;
- что относится к legacy;
- что является черновиками;
- что относится к эксплуатации;
- что является source of truth по коду и данным.

## Что считается архитектурой

На текущем этапе частью архитектуры считаются следующие папки и контуры:

- `backend`
- `bot`
- `support_bot`
- `dashboard`
- `dashboard/ui`
- `landing`
- `ops`
- `documentation`

Если коротко:

- `backend` — общее ядро;
- `bot`, `support_bot`, `landing`, `dashboard/ui` — интерфейсные контуры;
- `dashboard` — административный backend и часть текущего ядра;
- `ops` — эксплуатационный слой;
- `documentation` — канонический слой знаний о системе.

## Что не считается архитектурой

Следующее не должно считаться частью продуктовой архитектуры:

- `.git`
- `venv`
- `__pycache__`
- `dashboard/ui/.next`
- `dashboard/ui/node_modules`

Также не нужно включать в архитектурную карту локальные и случайные артефакты:

- временные markdown-черновики;
- рабочие картинки;
- локальные исследовательские заметки;
- автогенерируемые кеши и сборочные директории.

## Что считается legacy

На текущем этапе к legacy относятся:

- Jinja UI внутри `dashboard`;
- старые маршруты админки, которые живут ради совместимости;
- всё, что дублирует новый UI из `dashboard/ui`;
- старые или промежуточные описания структуры проекта, если они расходятся с текущей документацией.

Важно:

`dashboard` целиком не является legacy.  
Legacy в нём — это прежде всего старый UI-слой, а не backend/API.

## Что считается черновиками

На текущем этапе черновиками считаются:

- `docs` как папка рабочих заметок до миграции;
- временные аналитические файлы;
- промежуточные карты проекта;
- неканонические документы, созданные для обсуждения и переупаковки знаний.

Формальное правило:

- `documentation` — каноническая документация;
- `docs` — временные рабочие заметки до миграции.

## Что считается эксплуатацией

К эксплуатационному слою относятся:

- `ops`
- `backups`

Внутри `ops` находятся:

- `systemd`
- `nginx`
- env-файлы
- watchdog
- reminders
- служебные скрипты

Это не предметная логика продукта, но это часть продового контура.

## Source of truth по данным

Главный source of truth по данным — PostgreSQL.

Именно там живут основные сущности системы:

- пользователи;
- подписки и доступы;
- устройства;
- платежи;
- тикеты поддержки;
- административные сущности;
- часть служебных данных.

Важно:

- `dashboard/ui` не является source of truth;
- `landing` не является source of truth;
- Telegram-боты не являются отдельным хранилищем истины.

## Source of truth по коду

Source of truth по коду — текущий репозиторий, но читать его нужно с правильной иерархией доверия.

Приоритет такой:

1. Реальный runtime-код в `backend`, `bot`, `support_bot`, `dashboard`, `dashboard/ui`, `landing`, `ops`.
2. Каноническая документация в `documentation`.
3. Временные заметки в `docs` — только как черновой вспомогательный слой.

Если код и черновик расходятся, верить нужно коду и канонической документации, а не черновику.

## Спорная зона 1: dashboard vs dashboard/ui

Это главный переходный участок репозитория.

### Что остаётся в dashboard

В `dashboard` сейчас остаются:

- FastAPI backend админки;
- административный API;
- auth/session логика;
- backend для `dashboard/ui`;
- часть управленческой логики по пользователям, платежам, support, серверам и документации;
- legacy Jinja UI.

### Что уже переехало в dashboard/ui

В `dashboard/ui` уже находится новый пользовательский интерфейс админки:

- login / verify flow для команды;
- overview;
- users;
- servers;
- traffic;
- payments;
- support;
- knowledge;
- settings;
- общий app shell и frontend-навигация.

### Что считается legacy

Legacy в этой зоне — это:

- Jinja UI внутри `dashboard`;
- старые dashboard-маршруты, сохранённые ради совместимости;
- старые визуальные контуры, которые уже заменены `dashboard/ui`.

### Можно ли удалять Jinja UI сейчас

Пока рано удалять Jinja UI без отдельной проверки.

Причины:

- `dashboard` всё ещё содержит обязательный backend/API слой;
- в репозитории есть переходная mixed-схема;
- legacy-маршруты всё ещё участвуют в совместимости;
- перед удалением нужно отдельно проверить, не осталось ли экранов или flow, которые ещё не закрыты в `dashboard/ui`.

Формальное правило на сейчас:

- `dashboard/ui` — основной UI админки;
- `dashboard` — обязательный backend админки;
- Jinja UI внутри `dashboard` — legacy, но ещё не автоматически удаляемый слой.

## Спорная зона 2: docs vs documentation

В этой зоне должно быть формальное правило без двусмысленности:

- `documentation` — каноническая документация проекта;
- `docs` — временные рабочие заметки до миграции.

Следствие:

- новые системные документы нужно писать в `documentation`;
- `docs` не должен восприниматься как окончательный источник знания;
- полезные материалы из `docs` позже либо переносятся в `documentation`, либо удаляются.

## Спорная зона 3: backend vs логика внутри приложений

Важно зафиксировать честную картину:

ядро системы сейчас не собрано полностью в одной папке.

Оно живёт в:

- `backend`
- части логики в `bot`
- части логики в `dashboard`
- части логики в `support_bot`

Это не ошибка документации, а факт текущего состояния проекта.

Следствие:

- `backend` уже является ядром, но пока не единственным местом доменной логики;
- часть бизнес-правил всё ещё распределена по приложениям;
- при чтении репозитория нельзя автоматически считать, что вся доменная логика уже централизована.

Формула текущего состояния:

`backend` — центр ядра, но ядро пока частично размазано по связанным Python-контурам.

## Правило чтения репозитория

Если нужно быстро понять проект, читать его лучше в таком порядке:

1. `documentation/PROJECT_OVERVIEW.md`
2. `documentation/ARCHITECTURE.md`
3. `documentation/DOMAIN.md`
4. `documentation/RUNBOOK.md`
5. runtime-код в `backend`, `bot`, `support_bot`, `dashboard`, `dashboard/ui`, `landing`
6. `ops`
7. только потом временные заметки в `docs`

## Правила изменения кода

- Не считать `dashboard` полностью legacy.
- Не удалять Jinja UI без проверки покрытия и runtime-использования в `dashboard/ui`.
- Не переносить доменную логику из `bot`, `support_bot` или `dashboard` в `backend` без отдельного миграционного плана.
- Не менять runtime-пути, `systemd`-конфигурации или `nginx`-маршрутизацию без обновления `RUNBOOK.md`.
- Любое изменение source-of-truth моделей должно сопровождаться проверкой связанных flow: `bot`, `support_bot`, `dashboard`, `dashboard/ui`, payments и support.
- Если изменение затрагивает спорные зоны репозитория, сначала нужно проверить, не закреплено ли текущее правило в этом документе.

## Итоговое правило

При чтении любой папки нужно задать вопрос:

- это ядро?
- это интерфейс?
- это эксплуатация?
- это каноническая документация?
- это черновик или локальный артефакт?

Если папка попадает только в категорию черновиков, кешей или локальных артефактов, её нельзя считать архитектурой проекта.

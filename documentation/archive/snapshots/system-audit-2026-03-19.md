> Historical dated snapshot. Useful as a point-in-time audit for 2026-03-19, but not the primary source of current truth if newer canonical docs disagree.

1. По состоянию на 19 марта 2026 года проект Amonora работает на трёх VPS: `46.21.81.186` используется как backend/core, `213.108.20.34` используется как основная VPN-нода Германии, `185.88.37.71` используется как дополнительная VPN-нода Эстонии.
2. Основной вход для администрирования сейчас удобнее делать из Windows PowerShell, а не из WSL, потому что рабочий SSH-ключ уже привязан к Windows-профилю и команда входа на backend выглядит так: `ssh -i C:\Users\Skyfal\.ssh\id_ed25519 root@46.21.81.186`.
3. Из WSL прямой доступ ранее ломался из-за различий между ключами Windows и WSL, поэтому текущее практическое правило такое: если нужно зайти на сервер быстро и без сюрпризов, использовать Windows `ssh`, а не Linux-ключ из WSL.
4. Backend-сервер `46.21.81.186` сейчас несёт на себе `PostgreSQL`, основной Telegram-бот, support-бот, dashboard backend, Next.js dashboard v2 и `nginx`, то есть это единый core-сервер и главный control plane всего проекта.
5. По живым данным backend сейчас имеет `4 vCPU`, около `7.76 GB RAM`, диск около `76.45 GB`, занято примерно `9%`, хостнейм `hiplet-78194`, а внешнее геоопределение по IP показывает Стокгольм, Швецию.
6. Германия `213.108.20.34` сейчас является основной пользовательской VPN-нодой, домен для неё `ffconnect.amonoraconnect.com`, по IP-данным это Франкфурт, Германия, ASN `210644`, провайдер `Aeza International LTD`.
7. По живому snapshot Германия сейчас имеет `2 vCPU`, около `3.82 GB RAM`, диск около `58.96 GB`, занято около `6.5%`, uptime около двух суток, текущий health `healthy`, текущий ping по internal health около `23.4 ms`.
8. Эстония `185.88.37.71` сейчас является резервной и дополнительной VPN-нодой, домен для неё `connect.amonoraconnect.com`, геолокация определяется как Таллин, Эстония, ASN `41745`, провайдер по IP-данным `Baykov Ilya Sergeevich`, домен провайдера `fortis.host`.
9. По живому snapshot Эстония сейчас имеет `1 vCPU`, около `0.94 GB RAM`, диск около `8.65 GB`, занято около `55.1%`, uptime около двух с половиной суток, текущий health `healthy`, текущий ping по internal health около `10.4 ms`.
10. Отдельного standby-сервера, отдельного холодного резерва или выделенного disaster-recovery контура сейчас нет, поэтому фактический резерв по VPN обеспечивается только второй нодой, а backend остаётся single point of failure.
11. У каждого сервера сейчас по одному публичному IP, выделенных failover-IP, пулов адресов или автоматической ротации IP нет, а значит любая просадка IP-репутации ударяет сразу по конкретной стране и домену.
12. Основной сайт и dashboard живут на домене `amonoraconnect.com` и проходят через `nginx`, при этом веб-часть находится за Cloudflare, а сами VPN-контуры не сведены в отдельный Cloudflare-backed транспортный слой.
13. Новый основной dashboard уже переведён на корневые маршруты и работает по адресам `/login`, `/overview`, `/users`, `/servers`, `/traffic`, `/payments`, `/support`, `/settings`, `/knowledge`, а старые пути `/v2/*` и `/dashboard/*` в основном редиректятся в новый контур.
14. Dashboard v2 работает как Next.js-приложение на `127.0.0.1:3001`, backend JSON/API для него живёт на `127.0.0.1:8088`, а legacy/landing сервис обслуживается отдельно на `127.0.0.1:8090`.
15. Фактический конфиг systemd для нового dashboard лежит в `ops/systemd/amonora-dashboard-v2.service`, продовый env для него лежит в `ops/env/amonora-dashboard-v2.env`, а основной публичный reverse proxy настроен в `ops/nginx/amonora-dashboard.server.conf`.
16. База данных проекта уже не тестовая и не file-based: основное хранилище сейчас `PostgreSQL`, строка подключения собирается через `postgresql+asyncpg`, а локальные переменные указывают на `DB_HOST=localhost`, `DB_PORT=5432`, `DB_NAME=amonora_db`, `DB_USER=amonora`.
17. Support-бот, users, VPN-клиенты, платежи и dashboard-модели уже живут в Postgres, то есть `sqlite` или JSON-файлы не являются основным рабочим источником правды.
18. По текущей базе данных сейчас насчитывается `24` пользователя всего, из них `23` реальных без synthetic/test-префиксов, а распределение статусов выглядит так: `10 inactive`, `8 trial_active`, `2 paid_active`, `3 vip_active`.
19. По текущей базе VPN-клиентов сейчас есть `16` клиентских записей, из них `15` с протоколом `vless` и `1` с протоколом `trojan`.
20. По текущей базе payment records сейчас есть `3` записи, из них `1` подтверждённая и `2` ожидающих оплату, и все текущие записи относятся к `sbp_manual`, то есть основным рабочим способом платежа на сегодня является ручной СБП.
21. Единственный подтверждённый платёж сейчас — это `149 RUB` за тариф `1m`, а значит проект уже работает не только как тестовый стенд, а уже имеет минимум одного реального платного клиента.
22. Telegram Stars и Crypto Pay в коде предусмотрены, но по практическому состоянию проекта основным рабочим платежным потоком сейчас остаётся ручной СБП, а автоматическое массовое автопродление ещё нельзя считать завершённым продовым контуром.
23. Основной продуктовый VPN-стек сейчас построен на `3x-ui + Xray`, а backend обращается к панелям нод не по открытому интернету, а через локальные URL-туннели `127.0.0.1:12053` для Германии и `127.0.0.1:12054` для Эстонии.
24. Публичный `2053/tcp` продуктово не должен быть точкой интеграции backend-а, и текущая архитектура уже исходит из того, что панели `3x-ui` остаются внутренними.
25. На Германии сейчас боевые inbound таковы: `VLESS` на `443/tcp` с `security=reality`, `network=tcp` и `Trojan + TLS` на `8443/tcp`.
26. На Эстонии сейчас боевые inbound таковы: `VLESS` на `443/tcp` с `security=reality`, `network=tcp` и `Trojan + TLS` на `8443/tcp`.
27. На Эстонии в панели всё ещё висит legacy inbound `wireguard-51820` на `51820/udp`, но продуктово WireGuard уже не должен считаться главным пользовательским протоколом и воспринимается как технический хвост старой схемы.
28. Текущий Reality-профиль на обеих нодах использует один и тот же SNI `www.microsoft.com`, а shortId задан не одним значением, а набором из нескольких значений, причём набор сейчас одинаков на Германии и Эстонии.
29. Главные пользовательские протоколы в продукте сейчас — `VLESS` и `Trojan + TLS`, а `WireGuard` из пользовательского сценария уже в основном выведен и заменялся на Trojan.
30. Попытка перевести Эстонию на `VLESS + REALITY + XHTTP` реально делалась, но в продовых пользовательских тестах это привело к ошибкам `EOF` и `TLS handshake`, поэтому ноду откатили обратно на более совместимый `TCP`.
31. Это означает, что `XHTTP` сейчас нельзя считать готовым рабочим продовым стандартом для твоего стека клиентов, даже если он выглядит более привлекательным с точки зрения anti-DPI и теории маскировки.
32. Важная отдельная проблема, которая уже проявлялась, заключалась в том, что `3x-ui` однажды перевела VLESS-контур в новый PQ-режим с `mlkem...`, а реальные клиенты по выданным ссылкам это не переваривали, после чего inbound-ы пришлось возвращать в совместимый классический режим `decryption=none`.
33. Следовательно, любые будущие обновления панели и Xray нужно делать осторожно, потому что они могут тихо сломать уже рабочие ссылки даже без видимого падения серверов.
34. Мобильный onboarding в боте сейчас ориентирован на `Happ` для `VLESS` и `Trojan`, с логикой вопроса “установлен ли клиент”, ссылками в магазины и отдельной выдачей QR, но по факту стабильность зависит от конкретного приложения, и Happ уже показывал себя проблемным на advanced-сценариях.
35. Выдача ссылок в боте сейчас устроена так, что сама ссылка отдаётся сразу, а QR остаётся отдельной кнопкой, и это уже было специально доработано, чтобы не перегружать пользователя и не отправлять QR автоматически для VLESS/Trojan.
36. Имена конфигов и файлов уже делались ASCII-safe, чтобы кириллица не ломала импорт у клиентов, то есть человеческое имя устройства и техническое имя файла уже разведены аккуратнее, чем в самом начале.
37. Внутри продукта сейчас официально поддерживаются две страны, `de` и `ee`, а старый alias `nl` нормализуется в `de`, то есть продуктовая модель уже фактически строится на Германии и Эстонии.
38. По региональной логике Германия считается основной стабильной нодой, а Эстония — слабой резервной нодой и спец-маршрутом, а не массовым premium-кластером.
39. Для Эстонии в коде уже введены soft-limit правила по ресурсам: `CPU 85%`, `RAM 82%`, `disk 80%`, `load average 1.2`, что гораздо лучше прежнего грубого лимита “2 устройства”.
40. Watchdog сейчас уже умеет различать `down`, `overloaded` и `degraded`, запускается по timer unit каждую минуту, и рассылает сигналы админам через support-бота, а пользователям умеет отправлять сообщение о техработах при настоящем падении затронутой ноды.
41. Однако логику алертов нельзя считать окончательно калиброванной, потому что она уже не раз дорабатывалась, а кейс с Германией и `ping 111 ms` показал, что одинаковые жёсткие пороги не всегда отражают реальную норму конкретного VPS.
42. Dashboard уже видит три управляемых сервера как `Amonora Estonia Node`, `Amonora Core` и `Amonora Germany Primary`, а live-снапшоты в overview и servers уже строятся именно из этой управляемой тройки.
43. По текущему live-срезу dashboard видит примерно `14` active devices и `12` active users на Германии и `2` active devices и `1` active user на Эстонии, но эти цифры надо понимать как текущие агрегаты control plane и панели, а не как идеальную абсолютную истину “онлайн прямо сейчас”.
44. Новый dashboard v2 уже умеет overview, users, servers, traffic, payments, support, settings и knowledge, а overview уже содержит блок `Срез по тарифам`, который расположен на странице `overview` рядом с блоком `Структура базы и тарифов`.
45. Если пользователь не видит `Срез по тарифам` в браузере, это почти наверняка означает не отсутствие логики в коде, а либо старый кэш/старый bundle, либо продовый рассинхрон между исходником и live-build.
46. Сессии dashboard-а не вечные: в коде сейчас задан абсолютный session lifetime около `24` часов и idle-timeout `45` минут, так что админ после простоя действительно должен переаутентифицироваться.
47. Шаг подтверждения логина по Telegram-коду уже переводился на серверные `POST` маршруты `/auth/request-code` и `/auth/verify`, а не на чисто клиентский JS, потому что старый подход ронял cookies и вызывал ошибки вроде “Сессия истекла” сразу после ввода кода.
48. Код подтверждения живёт около `10` минут, старый код должен инвалидироваться при новом запросе, а логика выдачи в поддержку уже правилась несколько раз, так что этот контур рабочий, но его всё ещё нужно считать чувствительным к regressions.
49. Support-бот сейчас хранит тикеты и сообщения в Postgres, а не в transient памяти, с ограничением `50` сообщений на тикет и retention закрытых тикетов `90` дней, что уже даёт нормальную раннюю базу для SaaS-support сценария.
50. Product-wise support-логика и уведомления уже работают, но этот контур честно ещё не прошёл полный, спокойный финальный regression QA после всех быстрых правок на живом проекте.
51. Прямо сейчас polling в dashboard уже сделан умеренным и более безопасным для сервера: overview и traffic обновляются примерно раз в `20` секунд, support и notifications примерно раз в `12` секунд, payments примерно раз в `30` секунд, settings и knowledge без агрессивного polling.
52. Это уже лучше, чем ранние слишком частые запросы, но всё ещё polling-архитектура, а не вебсокеты, то есть при дальнейшем росте нагрузки этот слой надо будет мониторить отдельно.
53. Отдельной сильной стороной текущей реализации является то, что analytics в dashboard уже чистят synthetic/test users и не должны так сильно засоряться `manual_payment_`, `smoke_`, `test_`, `debug_`, `seed_` пользователями.
54. Слабой стороной проекта сейчас остаётся то, что ты уже вышел в ранний прод с реальными пользователями и первым платным клиентом, но при этом repository и deploy-процесс ещё не приведены к аккуратному зрелому состоянию.
55. Локальное `git status` сейчас сильно грязное и показывает множество изменённых файлов по `bot`, `dashboard`, `dashboard_v2`, `landing`, `ops`, `support_bot`, плюс untracked новые файлы, а это значит, что риск случайно откатить или потерять важные изменения уже высокий.
56. Поэтому на текущем этапе главная техническая опасность — не только баги продукта, но и процесс: без наведения порядка в git и нормального разбиения на коммиты/ветки следующий большой рефакторинг или новый VPN-контур может стать источником новых проблем сам по себе.
57. Секреты и живые токены в такой аудитовый markdown специально включать не нужно, потому что это было бы небезопасно для репозитория; в operational документацию стоит класть команды входа, домены, IP, роли серверов, сервисы и слабые места, но не рабочие пароли и токены в открытом виде.
58. Если говорить честно о проблемах, которые ещё не доведены до полного спокойного состояния, то это `XHTTP` как реальный прод-стек, полная regression-проверка dashboard после цепочки быстрых правок, финальная стабилизация support/notifications и приведение git-дерева в управляемое состояние.
59. Если говорить честно о том, что уже реально работает, то это разделённая инфраструктура backend/VPN, живой dashboard v2, Telegram login-flow, Postgres как основная БД, ручной СБП-платёж, базовый support-контур, VLESS и Trojan на двух нодах и live-monitoring через watchdog и dashboard snapshots.
60. Общая зрелость проекта на текущий момент — это ранний боевой сервис с реальными пользователями и первым платным клиентом, который уже даёт продуктовую ценность, но пока ещё не готов к безопасному масштабированию на большую аудиторию без отдельного этапа стабилизации VPN-архитектуры, полного QA и наведения порядка в процессе разработки и деплоя.

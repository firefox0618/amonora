# Amonora Status And Next Steps — 2026-03-19

> Historical dated planning snapshot. Useful as a point-in-time status note for 2026-03-19, but not the primary source of current truth if newer canonical docs disagree.

## Что уже сделано

1. Инфраструктура разделена на backend/core и отдельные VPN-ноды, что уже снижает риск, что рост ботов и dashboard будет мешать VPN-трафику.
2. Основной backend работает на отдельном сервере `46.21.81.186`, а Германия и Эстония используются как отдельные VPN-ноды.
3. Основной dashboard переведён на Next.js-контур и работает как главный интерфейс по корневым маршрутам сайта.
4. Старые маршруты `/dashboard/*` и `/v2/*` в основном сведены к редиректам в новый dashboard.
5. В проекте уже есть рабочий JSON API v2 для dashboard поверх текущего backend.
6. Логин в dashboard работает через существующую Telegram-схему, а не через отдельную новую auth-систему.
7. Сессии dashboard не вечные: уже заведён idle-timeout и абсолютный срок жизни сессии.
8. Основная база данных уже на PostgreSQL, а не на временных файлах.
9. Основной Telegram-бот работает с реальной базой пользователей и устройств.
10. Support-бот уже хранит обращения и сообщения в PostgreSQL.
11. В проекте уже есть рабочая логика support-тикетов, replies и хранения истории.
12. В dashboard уже есть страницы `Overview`, `Users`, `Servers`, `Traffic`, `Payments`, `Support`, `Settings`, `Knowledge`.
13. Overview уже показывает реальные KPI и не только мок-данные.
14. В overview уже выведен блок `Срез по тарифам` и пользовательская структура по планам.
15. Dashboard уже умеет глобальный поиск, notifications и polling-обновления по ключевым разделам.
16. Кнопка `Продлить тариф` в исходнике уже переведена на корректный `POST`, а не `GET`.
17. Иконка вкладки dashboard уже встроена в metadata как inline favicon, чтобы не зависеть от старых edge-404.
18. В коде уже есть реальный профиль администратора, роли и разграничение действий по ролям.
19. Платёжный слой уже живой, есть реальный подтверждённый СБП-платёж и связанная логика продления доступа.
20. В product-flow уже есть реальные тарифы и ручной payment review контур.
21. В проекте уже есть региональная логика по странам `de` и `ee`.
22. Основная продуктовая схема протоколов сейчас сведена к `VLESS` и `Trojan + TLS`.
23. WireGuard как основной пользовательский сценарий фактически уже вытеснялся из продукта.
24. В боте уже есть onboarding по созданию устройств с выбором ОС, протокола и страны.
25. Для `VLESS` и `Trojan` уже реализован Happ-based mobile onboarding с отдельной выдачей ссылки и QR по кнопке.
26. Имена конфигов и файлов уже нормализованы для более безопасного импорта без кириллицы.
27. Backend уже ходит к панелям `3x-ui` через SSH-туннели, а не напрямую по открытому HTTP через интернет.
28. Панели `3x-ui` не являются основной публичной точкой управления для backend-сервисов.
29. На Германии сейчас работают боевые inbound `VLESS + REALITY` и `Trojan + TLS`.
30. На Эстонии сейчас работают боевые inbound `VLESS + REALITY` и `Trojan + TLS`.
31. Для Эстонии уже введены soft-limit правила по `CPU`, `RAM`, `disk`, `load average` вместо грубого лимита по устройствам.
32. Watchdog уже умеет различать `down`, `degraded`, `overloaded`.
33. Watchdog уже шлёт уведомления админам и умеет уведомлять пользователей о техработах при реальном падении ноды.
34. Polling в dashboard уже был уменьшен до более безопасного уровня, чтобы не душить backend лишними запросами.
35. Analytics в dashboard уже фильтруют synthetic/test пользователей, чтобы не засорять реальные продуктовые метрики.
36. В проекте уже есть минимум один реальный платный клиент, то есть это уже не чистый тестовый стенд.

## Что ещё не закрыто или закрыто не до конца

1. `XHTTP` как реальный продовый transport для Эстонии пока не доведён до стабильного пользовательского качества.
2. Попытка перевода Эстонии на `XHTTP` уже давала реальные пользовательские ошибки `EOF` и `TLS handshake`.
3. Клиентский стек под `XHTTP` пока нельзя считать надёжно подтверждённым для массового прод-использования.
4. В панели и коде ещё остаются legacy-хвосты WireGuard, особенно на Эстонии.
5. Не все пользовательские действия dashboard прошли спокойный полный regression QA после серии быстрых фиксов.
6. Support-логика и уведомления уже рабочие, но ещё не проверены как полностью завершённый production-grade контур.
7. Платёжный слой сейчас по факту опирается в основном на ручной СБП, а не на полноценную автоматизированную систему подписок.
8. У проекта пока нет отдельного standby backend-сервера, и core остаётся single point of failure.
9. У проекта нет полноценного запасного IP-пула или балансировки по нодам.
10. Германия и Эстония используют близкую маскировочную схему, а не сильно разведённые anti-DPI профили.
11. Reality сейчас использует один и тот же SNI `www.microsoft.com` на обеих нодах, что снижает разнообразие маскировки.
12. Крупные будущие обновления `3x-ui/Xray` рискованны, потому что однажды уже ломали VLESS-ссылки через новый PQ-режим.
13. Прямые edge-роуты вроде `/favicon.svg` и `/brand-icon.svg` могли оставаться в старом состоянии через CDN-кэш, несмотря на рабочий inline favicon.
14. Полная визуальная и функциональная полировка dashboard ещё не завершена.
15. Support и notification UX ещё не выглядят как финальный polished SaaS-контур.
16. Эстония по железу не подходит для роли массовой premium-ноды.
17. Эстония остаётся слабой VPS: `1 vCPU`, `~1 GB RAM`, `~100 Mbps` по базовому профилю.
18. Нельзя честно обещать, что Эстония сейчас выдержит долгую массовую нагрузку как основной сервер.
19. Весь репозиторий сейчас в грязном состоянии: очень много изменённых и untracked файлов.
20. Пока git-дерево не приведено в порядок, любой следующий большой этап разработки рискованнее, чем должен быть.

## Что делать дальше в первую очередь

1. Зафиксировать текущее рабочее состояние репозитория: разложить изменения по смысловым блокам и перестать работать из сильно грязного дерева.
2. Провести полный regression QA по dashboard на живом проде: `Users`, `Payments`, `Support`, `Servers`, `Settings`, `Notifications`, `Knowledge`.
3. Прогнать повторно весь auth-flow dashboard: `login`, `request-code`, `verify`, idle-timeout, logout, role restrictions.
4. Отдельно перепроверить все мутирующие кнопки dashboard после последних быстрых деплоев.
5. Проверить полный пользовательский сценарий бота: создание устройства, повторная выдача ключа, удаление устройства, смена протокола, trial, продление.
6. Полностью дочистить legacy WireGuard, если он больше не нужен продуктово.
7. Не возвращать `XHTTP` в прод вслепую; если снова тестировать его, то только в отдельном controlled этапе и с понятным стеком клиентов.
8. Отдельно решить будущую VPN-архитектуру: оставаться на `3x-ui/Xray`, или выносить advanced-контур в отдельный Xray/Marzban-слой.
9. Если нужен premium/anti-DPI маршрут, сначала усилить железо Эстонии или поднять отдельную новую ноду, а уже потом продавать это как premium.
10. Не позиционировать Эстонию как “лучшая нода для всех”, пока она остаётся слабым VPS.
11. Выделить одну ноду как основную массовую, а вторую как резерв/спец-маршрут, и закрепить это в продукте, а не менять логику на лету.
12. После стабилизации VPN-части отдельно закрыть UI-полировку dashboard, чтобы не смешивать серьёзную сетевую архитектуру и косметические баги в одном проходе.
13. Привести в порядок production deploy-процесс: понятный rebuild, restart, smoke-check и фиксация результата после каждого выката.
14. Сделать короткий внутренний runbook: как зайти на сервер, как перезапустить сервисы, как проверить ноды, как отладить auth и payments.
15. После этого уже можно переходить к следующей фазе: масштабирование, выделенный premium-контур, новые ноды или новый VPN-control plane.

## Текущий практический вывод

1. Проект уже живой и реально полезный, потому что у него есть рабочая инфраструктура, реальные пользователи и уже минимум один реальный платёж.
2. Проект ещё не стабилизирован до состояния “масштабируем без страха”, потому что VPN-архитектура, regression QA и git-процесс ещё не доведены до спокойного зрелого уровня.
3. Самый правильный следующий шаг сейчас — не добавлять много новых сущностей подряд, а закрепить текущее рабочее состояние и снять технический риск с продового контура.

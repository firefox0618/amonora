# AMONORA — ARCHITECTURE V2 ROADMAP
Дата: 19 марта 2026

---

# ЦЕЛЬ

Этот документ показывает, как вырасти из текущего состояния
в устойчивый VPN-сервис без хаотичных прыжков и без ломки продa.

Главный принцип:

> Сначала стабилизация.  
> Потом управляемость.  
> Потом масштаб.  
> Потом усложнение архитектуры.

---

# 1. ТЕКУЩАЯ ТОЧКА

## Сейчас проект выглядит так:

- 1 core/backend server
- 2 VPN nodes
- PostgreSQL
- Telegram bot
- Support bot
- Next.js dashboard
- 3x-ui control plane
- manual SBP billing
- ранний anti-DPI стек
- реальные пользователи
- первый платёж

## Это уже не черновик.
Но это ещё не масштабируемая SaaS-платформа.

---

# 2. ГЛАВНЫЕ РИСКИ НА ТЕКУЩЕМ ЭТАПЕ

- backend single point of failure
- Germany как перегруженный основной контур
- Estonia как слабая резервная нода
- грязный git / слабая дисциплина деплоя
- нестабильность клиентской совместимости
- нет полноценного fallback-маршрута
- нет нормального DR / standby слоя

---

# 3. ПРИНЦИПЫ РОСТА

## Не делать:
❌ прыгать в сложную архитектуру раньше времени  
❌ строить enterprise-схему без пользователей  
❌ усложнять VPN-стек ради теории  
❌ внедрять новую панель без стабилизации текущей системы  

## Делать:
✔ стабилизировать текущее  
✔ повышать предсказуемость  
✔ готовить почву для масштабирования  
✔ менять архитектуру поэтапно  

---

# 4. ЭТАП V2.1 — СТАБИЛИЗАЦИЯ РАННЕГО ПРОДА

## Цель
Сделать так, чтобы проект перестал зависеть от хаотичных ручных правок.

## Что должно быть выполнено
- [ ] backup strategy
- [ ] rollback kit
- [ ] changelog дисциплина
- [ ] git cleanup
- [ ] regression checklist
- [ ] отдельные новые inbound для новых клиентов
- [ ] разделённые SNI / shortId по Germany и Estonia

## Результат
- ниже риск случайной поломки
- меньше хаоса
- выше предсказуемость

---

# 5. ЭТАП V2.2 — УПРАВЛЯЕМЫЙ VPN СЛОЙ

## Цель
Перестать иметь один “магический” рабочий профиль и перейти к управляемым профилям.

## Что делаем
- [ ] старые inbound оставляем как legacy-stable
- [ ] новые inbound выдаём только новым клиентам
- [ ] формируем profile matrix:
  - Germany stable
  - Estonia stable
  - Trojan fallback
- [ ] фиксируем, какие клиенты каким профилем реально работают

## Что это даёт
- можно анализировать совместимость
- можно точечно решать проблемы
- меньше паники “ничего не работает”

## Цель

## Нужно
- [ ] вести deploy log

- [ ] вести таблицу соответствия:
  - user

  - support issues

## Результат

- меньше повторяющихся ошибок

---

# 7. ЭТАП V2.4 — 100+ ПОЛЬЗОВАТЕЛЕЙ

## Условие входа в этап
- текущий прод стабилен
- нет хаотичных деплоев
- VPN профили понятны
- support поток предсказуем

## Что понадобится
- [ ] ещё минимум 1 нормальная VPN-нода
- [ ] усиление резервного контура
- [ ] честное разделение нод по назначению:
  - primary
  - fallback
  - experimental
- [ ] health-based распределение выдачи
- [ ] менее ручная выдача серверов

## Архитектура уровня 100+
```text
User → Bot/API → Backend → Node Selection → VPN Nodes → Internet
```

## Что меняется по мышлению
С этого этапа ты управляешь не “одним сервером”, а пулом нод.

---

# 8. ЭТАП V2.5 — 250–500 ПОЛЬЗОВАТЕЛЕЙ

## Условие входа
- 100+ пользователей без хаоса
- support не горит
- платежи более-менее стабилизированы
- есть понятный onboarding

## Что понадобится
- [ ] минимум 3–5 VPN нод
- [ ] разнос нод по ролям
- [ ] автоматический health scoring
- [ ] soft-routing по качеству нод
- [ ] более зрелый мониторинг
- [ ] анти-DPI профили по группам клиентов

## Желательная архитектура
- Core/backend отдельно
- VPN nodes пулом
- primary vs fallback routing
- отдельные новые профили для экспериментов
- отключение выдачи degraded нод

## Уже нужно думать о
- standby backup backend
- отдельном backup Postgres
- автоматизации deploy-проверок

---

# 9. ЭТАП V2.6 — 500+ ПОЛЬЗОВАТЕЛЕЙ

## Это уже другой класс проекта

С этого момента “ручной ранний прод” перестаёт быть приемлемым.

## Понадобится
- [ ] standby или failover core/backend
- [ ] регулярные backup restore test
- [ ] выделенный контроль за платежами
- [ ] более зрелая support-операционка
- [ ] строгий release process
- [ ] полноценный node inventory
- [ ] capacity planning

## Архитектурно
- control plane и data plane должны мыслиться отдельно
- node health должен влиять на выдачу
- degraded nodes должны исключаться автоматически
- новые конфиг-профили должны выкатываться постепенно

---

# 10. КОГДА ДУМАТЬ О СМЕНЕ ПАНЕЛИ / CONTROL PLANE

Сейчас:
- 3x-ui тебе ещё подходит
- менять панель прямо сейчас — лишний риск

Думать о следующем уровне стоит только когда:
- [ ] текущий прод стабилен
- [ ] есть нормальная дисциплина изменений
- [ ] есть понятный pain от текущей панели
- [ ] рост реально упирается в control plane

То есть не “из любопытства”, а по необходимости.

---

# 11. КОГДА ДУМАТЬ О СЛОЖНОМ ANTI-DPI

Не сейчас.

Сложный anti-DPI redesign нужен только если:
- базовый Reality + TCP уже стабилен
- текущая аудитория реально страдает от блокировок
- у тебя есть время на controlled rollout
- есть среда, где можно безопасно тестировать

Пока что правильнее:
- не ломать working stack
- улучшать сегментацию профилей
- наблюдать реальные кейсы клиентов

---

# 12. КОГДА АВТОМАТИЗИРОВАТЬ ПЛАТЕЖИ

Сейчас manual SBP ещё допустим,
потому что у тебя очень ранняя стадия.

Но автоматизация становится обязательной, когда:
- [ ] ручное подтверждение начинает тормозить рост
- [ ] количество оплат уже неудобно обрабатывать вручную
- [ ] support тратит слишком много времени на доступ/продление

До этого момента:
- ручной метод можно держать
- но flow должен быть понятным и без хаоса

# 13. ЧТО ДОЛЖНО БЫТЬ ГОТОВО ПЕРЕД РОСТОМ

## Минимальный набор зрелости
- [ ] backup + rollback
- [ ] clean deploy discipline
- [ ] понятный VPN profile strategy
- [ ] понятный support flow
- [ ] понятный billing flow
- [ ] node health visibility
- [ ] минимальная документация

Без этого рост только увеличивает хаос.

# 15. ТВОЯ РЕАЛЬНАЯ ДОРОЖНАЯ КАРТА В ОДНОЙ ФРАЗЕ

> Сейчас тебе не нужна “крутая архитектура”.  
> Тебе нужна стабильная управляемая архитектура, которая не развалится от следующего десятка пользователей.

---

# 16. KPI ПО ЭТАПАМ

## Этап стабилизации
Успех:
- нет случайных поломок
- есть backup
- есть rollback
- новые inbound тестируются безопасно

## Этап 100+
Успех:
- несколько нод
- понятная выдача
- support справляется
- нет паники от каждой жалобы

## Этап 500+
Успех:
- control plane зрелый
- процессы зрелые
- backend не одинокий SPOF
- рост не ломает качество

---

# 17. ФИНАЛЬНОЕ ПРАВИЛО

> Масштабировать можно только то, что уже стабильно работает.
> Всё остальное — не рост, а размножение проблем.

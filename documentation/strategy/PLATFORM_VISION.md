# PLATFORM_VISION

## Назначение

Этот документ фиксирует целевое видение `Amonora` как экосистемы.

Важно:

- это не описание текущего production state;
- это не замена `PROJECT_OVERVIEW.md` и не замена `ARCHITECTURE.md`;
- это стратегический слой, который помогает понимать, куда экосистема должна расти после стабилизации текущего ядра.

## Базовая идея

`Amonora` — это не только VPN.

Целевая модель:

- единое ядро;
- несколько продуктовых модулей;
- единые пользовательские и административные поверхности;
- рост от одного рабочего продукта к экосистеме.

## Общая модель

Упрощённо экосистема выглядит так:

```text
Amonora Core
  -> Access
  -> Automations
  -> Data
  -> AI
```

Или в продуктовой логике:

```text
Users
  -> Bot / Web / Admin surfaces
  -> Core
  -> Product modules
  -> Revenue / retention / expansion
```

## Ядро экосистемы

Общее ядро должно обеспечивать:

- управление пользователями;
- авторизацию;
- биллинг;
- доступ и entitlement-логику;
- хранение данных;
- административное управление;
- support и audit слой.

На текущем этапе прообраз этого ядра уже виден в:

- `backend`
- `bot`
- `dashboard`
- `support_bot`
- PostgreSQL

Но в стратегическом смысле это именно ядро экосистемы, а не только VPN-проекта.

## Модули экосистемы

### 1. Access

Текущий основной рабочий модуль.

Включает:

- VPN
- будущие соседние access-сценарии вроде DNS / Proxy

Ценность:

- стабильный доступ;
- обход блокировок;
- защита пользовательского трафика.

### 2. Automations

Следующий приоритетный модуль роста.

Включает:

- Telegram automation;
- n8n workflows;
- бизнес-боты;
- автоответы;
- интеграции.

Ценность:

- экономия времени;
- автоматизация рутинных процессов;
- быстрый go-to-market для нового соседнего продукта.

### 3. Data

Будущий модуль.

Включает:

- мониторинг;
- парсинг;
- сбор данных;
- алерты.

Ценность:

- данные как продукт;
- data-driven сценарии;
- основа для будущих AI и automation workflows.

### 4. AI

Будущий модуль более позднего этапа.

Включает:

- AI-ботов;
- AI-помощников;
- генерацию;
- анализ;
- support и classification workflows.

Ценность:

- интеллектуализация экосистемы;
- повышение LTV и расширение продуктового слоя.

## Принцип роста

Ключевое правило экосистемы:

не запускать всё сразу.

Стратегически правильная последовательность:

1. Стабилизировать текущий Access/VPN-контур.
2. Усилить control plane и ops-дисциплину.
3. Добавить Automations как ближайший соседний продукт.
4. Только потом расширяться в Data и AI.

## Правила стратегии

- не ломать текущий VPN-продукт ради будущих модулей;
- не строить сложную платформу раньше времени;
- не распыляться на несколько новых продуктов одновременно;
- использовать текущее ядро как базу для новых модулей;
- расширять экосистему по мере подтверждения спроса и устойчивости.

## Бизнес-логика экосистемы

Сильная модель экосистемы выглядит так:

- Access создаёт доверие и первую пользовательскую базу;
- Automations дают новый денежный трек;
- Data усиливает ценность и наблюдаемость;
- AI добавляет следующий слой автоматизации и интеллекта.

Идея не в том, чтобы “уйти из VPN”, а в том, чтобы:

- начать с работающего access-продукта;
- вырастить вокруг него платформенный слой;
- строить модули, которые усиливают друг друга.

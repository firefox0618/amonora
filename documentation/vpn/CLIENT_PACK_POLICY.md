# Client Pack Policy

## Goal

Official client packs must exist as first-class deliverables, not only as prose.

For this stage, packs are maintained for:

- `v2rayNG`
- `Nekoray`
- `Streisand`

## Region policy

- preferred modern route: `Denmark`
- stable compatibility route: `Germany`
- reserve/testing route: `Estonia` is not user-facing

## Pack set

### Denmark preferred split tunnel

- `documentation/vpn/client-packs/v2rayng-split-tunnel.json`
- `documentation/vpn/client-packs/nekoray-split-tunnel.json`
- `documentation/vpn/client-packs/streisand-split-tunnel.json`

### Full-tunnel fallback

- `documentation/vpn/client-packs/v2rayng-full-tunnel.json`
- `documentation/vpn/client-packs/nekoray-full-tunnel.json`
- `documentation/vpn/client-packs/streisand-full-tunnel.json`

## Pack formats

Official client packs are delivered as:

- JSON import artifacts
- QR-code references for mobile onboarding
- text snippets with MTU and routing guidance

## MTU policy

- default: `1400`
- troubleshooting fallback: `1420`

Rules:

- if the client supports MTU in TUN/profile settings, store `1400` directly;
- if the client does not support imported MTU, show `1400` in the import instructions;
- the same MTU guidance also applies to `Germany` compatibility packs; there is no separate `1500` default policy for the current rollout;
- `1420` is only the fallback, not the default product recommendation.

## Split-tunnel policy

### Direct

Domestic destinations should stay direct:

- `geoip:ru`
- `geoip:private`
- `geosite:category-ru`
- `geosite:yandex`
- `geosite:vk`
- explicit domestic domains where needed

Current Xray-compatible routing block is built with:

- `domainStrategy = IPIfNonMatch`
- direct rule for domestic domains
- direct rule for `geoip:ru` and `geoip:private`
- blocked rule for `bittorrent`
- final `tcp,udp -> proxy` catch-all rule

### Proxy

Blocked/global destinations should go through VPN:

- `YouTube`
- `Google`
- `X / Twitter`
- `Instagram`
- other global destinations that are either blocked or better served through the VPN route

### Blocked

- `bittorrent`

## Client notes

### v2rayNG

- intended as the Android-first routing pack
- expects Xray-style routing semantics
- MTU should be set to `1400` where TUN/profile options expose it

### Nekoray

- intended as the desktop routing pack
- compatible with Xray/sing-box style routing concepts
- MTU should be set to `1400` in TUN/TAP or equivalent profile settings

### Streisand

- intended as the iOS/macOS-oriented routing pack reference for the supported rollout
- use `1400` MTU when the client build exposes it
- if MTU cannot be imported from a profile, keep it in operator/user import instructions

## Important boundary

These pack artifacts intentionally do not contain per-user credentials.

They are:

- routing policy artifacts
- MTU policy artifacts
- region/fallback policy artifacts

Per-user VLESS credentials are still generated by the device provisioning flow.

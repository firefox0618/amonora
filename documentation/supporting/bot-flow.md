## Основная роль бота

Бот управляет доступом, устройствами, оплатой и входом в поддержку.

## Главные разделы

- `Личный кабинет`
- `Устройства`
- `Купить`
- `Поддержка`
- `Информация`
- `Реферальная система`

На текущем этапе главные user-facing экраны бота собраны в более плотный визуальный формат с единым разделителем `━━━━━━━━━━━━━━━━━━━`.

Ключевой принцип редизайна:

- это redesign главных экранов, а не refactor доменной логики;
- статусы, access и device-domain semantics не меняются;
- старые текстовые aliases (`Канал`, `Информация`, `Рефералы`) сохраняются ради совместимости.

## Доступ

### Пробный доступ

Пользователь может получить `пробный доступ` после входа в бот и прохождения базового сценария.

Важно:

- в интерфейсе используется именно формулировка `пробный доступ`
- доступ должен проверяться не только в `/start`, но и во всех рабочих сценариях

### Платный доступ

Активный доступ считается валидным, если:

- пробный период ещё не закончился
- или активна платная подписка

## Устройства

Устройство в системе должно быть человеко-понятной сущностью, а не просто UUID.

Сейчас для устройства важны:

- имя
- ОС
- протокол
- страна
- дата создания
- статус доступа

В пользовательском списке устройств сейчас используется presentation-level модель:

- экран показывает, сколько устройств создано из `3`;
- выше inline-кнопок рендерится компактный список устройств;
- для текущего scope все устройства на активном access-экране отображаются как `🟢 Активно`;
- отдельный per-device health/runtime state на этом экране пока не вводится.

### Сценарий создания

1. Пользователь открывает раздел `Устройства`
2. Нажимает `Создать устройство`
3. Вводит имя
4. Выбирает ОС
5. Выбирает протокол
6. Выбирает страну
7. Получает ключ для `VLESS` / `Trojan`

## Протоколы и страны

Текущий рабочий набор по UX:

- `VLESS`
- `Trojan`

Текущие приоритетные страны:

- `Германия`
- `Эстония`

## Оплата

Интерфейс оплаты уже должен поддерживать несколько каналов:

- `Telegram Stars`
- `Ручная СБП`
- `Ручная крипта`

Текущий боевой принцип:

- `Telegram Stars` — нативный метод внутри Telegram;
- `СБП` и `крипта` — временный ручной контур с подтверждением администратора;
- auto `Crypto Bot` можно держать в коде за feature flag, но не включать в пользовательский поток без отдельной проверки.

Ручной payment flow:

1. Пользователь выбирает тариф.
2. Выбирает `СБП` или `Крипта`.
3. Бот показывает реквизиты и номер заявки.
4. После нажатия `Я оплатил` заявка уходит в очередь проверки.
5. Админ подтверждает или отклоняет её в `@amonora_control_bot` или в Панели управления.
6. Только после подтверждения доступ продлевается.

Дополнение:

- в RUB-flow сначала автоматически используется доступный `Баланс`;
- если Баланса хватает полностью, бот активирует доступ без внешнего платежа;
- `Telegram Stars` не смешиваются с Балансом.

## Реферальная система

Реферальный поток должен быть реальным, а не декоративным.

Базовые элементы:

- реферальная ссылка
- `50 р` за квалифицированного оплаченного реферала
- учёт переходов по приглашению
- отдельный счётчик друзей с начисленным бонусом
- отображение начислено всего / доступно сейчас

Экран и entrypoint в основном боте теперь называются именно `🎁 Реферальная система`.

## Поддержка

Кнопка поддержки из основного бота должна вести в отдельного `@amonora_support_bot`.

Support intro-screen в основном боте теперь остаётся коротким:

- перечень типовых проблем;
- обещание разумного времени ответа;
- одна основная кнопка перехода в support bot.

## Информация

В основном боте теперь есть отдельный раздел `📚 Информация`, который заменяет старую отдельную кнопку `Канал`.

Он работает как компактный in-bot hub и даёт доступ к:

- `Инструкциям`
- `FAQ`
- `Документам`
- Telegram-каналу
- support-ссылке

## UX-принцип

Бот не должен засорять историю лишними сообщениями. Где возможно, интерфейс должен:

- обновлять текущее сообщение
- показывать уже обновлённое состояние экрана
- избегать мусора из промежуточных системных ответов

# Отчёт по оптимизации Amonora

_Дата: 16 марта 2026_

## Что было улучшено

### 1. Оптимизация dashboard-runtime

- Добавлен короткий runtime-cache для тяжёлых сводок:
  - `overview_metrics`
  - `get_service_statuses`
  - `get_server_snapshots`
  - `xui_summary`
- Это снижает количество повторных обращений к:
  - `systemctl`
  - `journalctl`
  - `3x-ui`
  - системным метрикам сервера
- Кэш автоматически сбрасывается после критичных действий:
  - создание и удаление устройства
  - выдача пробного доступа
  - продление подписки
  - блокировка пользователя
  - обновление статуса сервера
  - действия над сервисами

### 2. Отчёты внутри документации

- Во вкладку `Документация` добавлена кнопка генерации операционного отчёта.
- Новый отчёт создаётся как Markdown-файл и сразу становится доступным в панели.
- Отчёт включает:
  - серверный контур
  - загрузку и ping
  - статусы сервисов
  - статистику пользователей
  - устройства и протоколы
  - платежи
  - бюджетные ориентиры
  - рекомендации по действиям

### 3. Исправление ошибки документации

- Исправлен `Internal Server Error` во вкладке `Документация`.
- Причина была в Jinja-шаблоне: использовался `section.items`, из-за чего шаблон получал встроенный метод словаря вместо списка документов.
- Исправлено на явный доступ к данным через `section["items"]`.

## Что появилось в интерфейсе

- Кнопка `Сгенерировать отчёт` во вкладке `Документация`.
- Кнопка `Последний отчёт`, если автоотчёт уже создан.
- Для сгенерированного отчёта отображается отдельный источник:
  - `Сгенерированный отчёт`
- В карточке документации теперь видно состояние автоотчёта.

## Бюджетный блок

Для автоотчётов добавлены переменные окружения:

- `DASHBOARD_MONTHLY_SERVER_BUDGET_RUB`
- `DASHBOARD_MONTHLY_OPERATIONS_BUDGET_RUB`
- `DASHBOARD_MONTHLY_TARGET_REVENUE_RUB`

Если они не заполнены, отчёт это честно показывает и помечает бюджет как не настроенный.

## Проверки и тесты

### Локально

Пройдено:

- `python -m compileall dashboard bot backend support_bot`
- импорт:
  - `dashboard.main`
  - `dashboard.services`
- smoke-check:
  - `get_documentation_page(None)`

Ограничение локального smoke-теста:

- полноценная генерация отчёта локально не выполнялась без активного доступа к Postgres, потому что локальная среда в этот момент не имела подключения к `127.0.0.1:5432`.

## Итог

Текущая оптимизация улучшила не только скорость дашборда, но и операционную управляемость проекта:

- меньше тяжёлых повторных вызовов
- быстрее открываются живые сводки
- появился единый генератор операционных отчётов
- документация снова стабильна и не падает
- вкладка `Документация` стала реальным центром знаний и отчётности

## Следующий шаг

Логично развивать это дальше в три стороны:

1. Исторические графики по CPU/RAM/Disk и задержкам.
2. Расширенный финансовый блок с расходами по серверам и каналам оплаты.
3. Полноценный планировщик отчётов: daily / weekly / monthly.

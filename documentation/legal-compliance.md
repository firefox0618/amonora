# Юридические и регуляторные риски Amonora (Россия)

## Важно

Этот документ не является юридическим заключением и не может гарантировать:

- что проект "точно не заблокируют";
- что "точно не будет претензий" со стороны регуляторов, платформ или платёжных провайдеров;
- что текущая модель уже полностью безопасна для масштабирования.

Он нужен как практический risk memo по текущему состоянию проекта `Amonora`, чтобы снизить риск блокировок, претензий и продуктовых ошибок до отдельной консультации с профильным юристом по РФ.

## Короткий вывод

На текущем этапе `Amonora` можно сделать заметно безопаснее, если соблюдать 5 правил:

1. Не позиционировать продукт как средство обхода блокировок, доступа к запрещённым ресурсам или "невидимости для РКН".
2. Держать платежи внутри Telegram для цифрового доступа только через `Telegram Stars`.
3. Формализовать обработку персональных данных: политика, уведомление, сроки хранения, проверка локализации.
4. Вынести внешние способы оплаты (`СБП`, электронный кошелёк, крипта) в отдельный web checkout после отдельной правовой проверки.
5. Не публиковать в канале, боте и документации формулировки вроде "обход блокировок", "доступ к заблокированным сайтам", "обход РКН", "разблокировка запрещённых ресурсов".

## Что уже выглядит относительно безопасно

По текущему репозиторию проведена выборочная проверка текстов и логики:

- в пользовательских текстах бота не найдено явного позиционирования как "обход блокировок";
- используется нейтральная формулировка `пробный доступ`, а не агрессивный маркетинг про "разблокировку";
- основной публичный UX сейчас описывает сервис как VPN с управлением устройствами, а не как инструмент доступа к запрещённым ресурсам;
- support-бот и дашборд уже позволяют меньше светить чувствительные данные в системных логах.

Это хороший старт, но его недостаточно для публичного роста.

## Нормативная и платформенная база, на которую стоит опираться

### 1. Telegram Stars

Telegram прямо указывает, что цифровые товары и услуги внутри Telegram apps должны оплачиваться через `Telegram Stars (XTR)`.

Практический вывод для `Amonora`:

- продажа доступа к VPN внутри бота должна идти через `Stars`;
- `provider_token` для `XTR` в `sendInvoice` должен быть опущен;
- внешний `СБП`, электронный кошелёк или крипта не должны быть основной in-app оплатой внутри Telegram-бота.

Источники:

- [Telegram Bot Payments for Digital Goods and Services](https://core.telegram.org/bots/payments-stars)
- [Telegram Bot API changelog: provider_token for XTR should be omitted](https://core.telegram.org/bots/api-changelog)

### 2. Персональные данные и локализация

На официальном портале Роскомнадзора прямо указано:

- уведомление в РКН обычно подаётся **до начала обработки** персональных данных;
- при сборе персональных данных, в том числе через Интернет, запись, систематизация, накопление, хранение, уточнение и извлечение должны осуществляться с использованием баз данных на территории РФ.

Практический вывод для `Amonora`:

- если сервис собирает данные пользователей из РФ, нужно отдельно проверить, где фактически расположена основная база данных;
- нельзя считать Telegram `username` и историю обращений "неперсональными" данными;
- текущая серверная схема должна быть проверена именно на предмет **первичной записи и хранения ПДн**.

Источники:

- [РКН: уведомление об обработке персональных данных подаётся до начала обработки](https://pd.rkn.gov.ru/operators-registry/operators-list/?id=4963)
- [РКН: при сборе ПДн через Интернет использовать базы данных на территории РФ](https://pd.rkn.gov.ru/operators-registry/operators-list/?id=5239)

### 3. Информация о способах доступа к ограниченным ресурсам

После изменений законодательства в этой сфере нельзя исходить из логики "VPN сам по себе нейтрален, значит можно рекламировать его как угодно". В практическом смысле это означает:

- нельзя строить маркетинг вокруг "обхода блокировок";
- нельзя делать FAQ/инструкции в стиле "как зайти на заблокированный ресурс";
- нельзя использовать формулировки "РКН не увидит", "обход запретов", "доступ к запрещённым сайтам".

Для `Amonora` безопаснее позиционировать продукт как:

- защищённое соединение;
- privacy / secure networking;
- удалённый доступ;
- защита публичных Wi-Fi;
- стабильное подключение устройств.

Источники:

- [Официальная публикация Федерального закона от 31.07.2023 № 406-ФЗ](https://publication.pravo.gov.ru/document/0001202307310001)
- [Официальная публикация акта Роскомнадзора по взаимодействию по ст. 15.8](https://publication.pravo.gov.ru/document/0001202504070015)

## Что найдено по текущему проекту

### Позитивные наблюдения

- В коде и документации нет явных публичных фраз про "обход блокировок".
- Канал `amonora_vpn` пока пустой, значит самый рискованный контент ещё можно сразу выстроить правильно.
- Основная модель оплаты уже построена вокруг `Telegram Stars`.
- Дашборд и support-бот позволяют централизовать контроль без лишней ручной пересылки чувствительных данных.

### Замечания и красные флаги

1. В проекте пока нет отдельного публичного документа:
   - политики обработки персональных данных;
   - оферты / пользовательского соглашения;
   - правил поддержки и возврата.

2. Support-бот хранит историю переписки в `PostgreSQL`, включая previews сообщений. Это чувствительные пользовательские данные.

3. В интерфейсе бота `СБП` и `крипта` сейчас реализованы как ручные заявки с подтверждением администратора. Это безопаснее, чем полноценная in-app продажа цифрового доступа не через `Stars`, но всё равно требует отдельного процесса поддержки и правовой оценки.

4. Нужно отдельно проверить реальное местонахождение основной БД и соответствие требованиям локализации, если сервис ориентирован на пользователей из РФ.

5. Нужен отдельный review по договорной модели:
   - кто продаёт доступ;
   - кто принимает деньги;
   - кто отвечает по поддержке;
   - какие контакты и реквизиты публикуются клиенту.

## Что нужно сделать до публичного масштабирования

### Обязательный минимум

1. Подготовить публичную `Политику обработки персональных данных`.
2. Подготовить `Оферту / Пользовательское соглашение`.
3. Подготовить `Политику возврата и поддержки`.
4. Проверить обязанность уведомления РКН и локализацию ПДн.
5. Провести отдельный legal review по модели продаж VPN-доступа в РФ.

### По продукту и контенту

1. Зафиксировать список запрещённых формулировок для бота, канала, лендинга и операторов.
2. Не публиковать инструкции вида "как открыть X заблокированный сервис".
3. Не использовать обещания вроде:
   - "не заблокируют";
   - "РКН ничего не увидит";
   - "100% анонимность";
   - "полный обход всех ограничений".

### По данным и логам

1. Ограничить срок хранения истории поддержки.
2. Не хранить лишние сообщения в системных логах.
3. Ввести журнал доступа к sensitive-данным в дашборде.
4. Разделить операционные логи и пользовательские данные.

## Рекомендация по платежам

### Что делать внутри Telegram

Для продажи цифрового доступа внутри Telegram-бота безопаснее держать только:

- `Telegram Stars`

### Что делать вне Telegram

Если нужен `СБП`, электронный кошелёк или крипта, то правильнее строить это так:

- отдельная web-страница оплаты;
- провайдер с webhook и нормальной сверкой статусов;
- отдельные правовые документы для внешней оплаты;
- отдельный процесс поддержки и возвратов.

### Что не стоит делать

Не стоит запускать оплату как:

- перевод на личный электронный кошелёк "и пришлите скрин";
- перевод на личную карту;
- ручной перевод на личный криптокошелёк без invoice/webhook/учёта.

Это не убирает вопросы, а наоборот создаёт их:

- по спорам;
- по учёту;
- по ошибкам идентификации платежа;
- по комплаенсу платёжных партнёров;
- по пользовательским жалобам.

## Практическое решение для Amonora на текущем этапе

Наиболее безопасная промежуточная схема сейчас такая:

1. В Telegram-боте оставить `Telegram Stars` как основной нативный платёж.
2. `СБП` и крипту держать только как временный ручной контур с подтверждением администратора, без обещаний мгновенной in-app оплаты.
3. Если нужен внешний платёжный контур:
   - сделать отдельный сайт / checkout;
   - подключить провайдера;
   - провести отдельную legal/tax проверку.

## Финальный вывод

У `Amonora` есть рабочая техническая база, и текущие тексты проекта уже ближе к безопасному позиционированию, чем многие типовые VPN-боты. Но до состояния "минимально безопасно для масштабирования" проекту всё ещё нужны:

- правовые документы;
- проверка ПДн и локализации;
- аккуратная платёжная архитектура;
- дисциплина контента;
- отдельный legal review по РФ.

Гарантию отсутствия блокировок или претензий дать нельзя. Можно только системно уменьшить риск, и именно на это должен быть ориентирован следующий этап.

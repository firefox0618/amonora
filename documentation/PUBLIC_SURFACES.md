# PUBLIC SURFACES

## Зачем нужен этот документ

Этот документ фиксирует публичные точки присутствия текущего продукта `Amonora`.

Он нужен для того, чтобы отдельно от архитектуры держать в одном месте:

- внешние точки входа;
- публичные каналы коммуникации;
- нерешённые вопросы вроде канала отзывов.

## Основные публичные точки входа

На текущем этапе у продукта есть такие внешние поверхности:

- `@amonora_bot` — основной продуктовый бот для клиентов;
- `@amonora_v_2_0_bot` — изолированный тестовый бот для проверки нового UX и монетизационных сценариев до переноса в основной клиентский контур;
- `@amonora_support_bot` — отдельный бот поддержки;
- `amonoraconnect.com` — публичный сайт и витрина продукта;
- `www.amonoraconnect.com` — канонический пользовательский web-entry для сайта и панели управления;
- `client.amonoraconnect.com/<token>` — персональная единая ссылка на подписку: браузер получает web-страницу, а Happ импортирует feed с того же URL;
- `client.amonoraconnect.com/sub/<token>` — compatibility feed-route для той же подписки, который больше не считается основным user-facing URL;
- `client.amonoraconnect.com/happ/add?sub=<encoded-url>` — Amonora-owned HTTPS wrapper для Happ deep-link открытия той же подписки без внешнего `lavivas`-прокси;
- `POST amonoraconnect.com/bridge/access` — публичный ограниченный bridge-access маршрут для выдачи бесплатного временного ключа подключения на `24 часа`, если пользователю сначала нужно открыть Telegram;
- `POST amonoraconnect.com/webhooks/platega/{secret}` — provider callback для auto `СБП/крипта` через `Platega`;
- `amonoraconnect.com/manual` — скрытая пользовательская инструкция, доступная только по прямой ссылке;
- `amonoraconnect@yandex.ru` — публичный email для обратной связи, претензий и запросов по персональным данным;
- [`@amonora_new`](https://t.me/amonora_new) — Telegram-канал проекта.

Важно:

- `@amonora_control_bot` не относится к публичным поверхностям продукта;
- это внутренний admin-only контур для системных уведомлений и operational review.
- `@amonora_v_2_0_bot` не является каноническим входом в продукт и не должен подменять `@amonora_bot`; это отдельная живая тестовая поверхность для controlled rollout новых пользовательских экранов и payment/bonus сценариев.
- `https://amonoraconnect.com/n8n/` может быть reachable на том же домене, но это тоже не продуктовая публичная поверхность.
- `n8n`-панель считается operator-only маршрутом: она закрыта outer `HTTP Basic Auth` и внутренней `n8n` auth/setup, и не должна использоваться как пользовательский вход, support-канал или onboarding surface.
- `https://grafana.amonoraconnect.com/` также может быть reachable как отдельный operator-only hostname, но это тоже не продуктовая публичная поверхность.
- `Grafana` считается operator-only маршрутом: она должна быть закрыта outer `HTTP Basic Auth`, использовать read-only datasource через tunnel к core PostgreSQL и не должна становиться пользовательским или support-входом.
- tokenized surface `client.amonoraconnect.com/<token>` считается публичной продуктовой поверхностью, но не общим открытым landing: это user-scoped link, которая должна выдаваться из `@amonora_bot`, не индексироваться как обычный сайт и не превращаться в отдельный web-cabinet или web-cashier.

Если говорить о главном пользовательском входе, то это `@amonora_bot`.

Текущее уточнение по web-entry:

- из-за жалоб на нестабильную доступность apex-host в части мобильных сетей канонический пользовательский веб-вход сейчас нужно считать `https://www.amonoraconnect.com`;
- landing legal/manual pages должны канонизироваться на `www`;
- при этом существующие POST-seams `bridge/access` и `webhooks/platega` остаются совместимыми на текущем origin-host и не должны ломаться из-за user-facing host canonicalization.

При этом публичный сайт теперь играет роль резервного входа:

- если Telegram недоступен напрямую, пользователь может запросить на landing бесплатный bridge-ключ на `1 день`;
- после подключения через этот временный ключ пользователь должен перейти в `@amonora_bot` и продолжить основной сценарий уже там;
- активный landing-flow не должен вести пользователя в отдельную веб-оплату, а должен возвращать его в Telegram-first контур.
- auto-checkout для `Platega` создаётся из `@amonora_bot`; landing только принимает callback и не становится отдельной web-cashier поверхностью.
- `bridge/access` теперь дополнительно ограничен не только app-side cooldown, но и edge rate-limit на `nginx`, чтобы этот public seam не оставался только in-memory защитой.
- legacy `Crypto Pay` webhook больше не считается активной публичной поверхностью: route остаётся в коде только как выключенный rollback seam и по умолчанию отвечает `410`.
- legacy `POST /vpn/activate` больше не считается рабочей публичной или скрытой product surface: compatibility route может кратковременно отвечать `410 Gone`, но продуктовый EE-activation flow выведен из эксплуатации.

Внутри самого `@amonora_bot` пользовательский entry shell теперь собран вокруг шести основных экранов:

- `Личный кабинет`
- `Устройства`
- `Купить`
- `Поддержка`
- `Информация`
- `Реферальная система`

Текущее уточнение:

- кнопка `Реферальная система` остаётся частью shell и теперь снова открывает живой экран с персональной ссылкой, балансом, статистикой и progress-геймификацией.
- выбор способа подключения в основном боте теперь подаётся как последовательность `страна -> режим`, где пользователь видит режимы `Стабильный / Мобильный / Резерв`, а не прямой выбор технического протокола.
- режим `Мобильный` теперь доступен в user-facing UX для Germany и Denmark как отдельный маршрут для сетей, где обычное подключение проходит нестабильно; сам transport по-прежнему скрыт за продуктовым названием режима.
- deep-link payload `post_<token>` теперь зарезервирован под переходы из публичного канала в `@amonora_bot`; это не отдельная новая поверхность, а tracked entry в тот же основной бот.
- в `Личном кабинете` основного бота теперь есть кнопка `Единая ссылка`, которая ведёт на user-level public subscription page и не заменяет текущий device-flow.

## Единая ссылка на подписку

Новый public surface `client.amonoraconnect.com` обслуживает user-level страницу подписки.

Текущие правила этого surface:

- canonical user-facing URL теперь один: `/<token>`;
- если `/<token>` открывает браузер, пользователь видит статус подписки, срок, install-flow и блок `QR + copy`;
- если тот же `/<token>` открывает Happ, `landing` по client-aware headers/UA отдаёт subscription feed вместо HTML;
- compatibility route `/sub/<token>` остаётся живым только как fallback seam и не должен раздаваться пользователю как основная ссылка;
- route `/happ/add?sub=...` теперь служит как Amonora-owned wrapper для Telegram/web переходов в Happ: он принимает только `client.amonoraconnect.com/<token>` или `/sub/<token>` на том же host, пытается открыть `happ://add/...` и оставляет fallback `Открыть Happ / Скопировать ссылку / Открыть страницу подписки`;
- server-side bind привязывает Happ-импорт к одному из слотов `1..N`, где `N` совпадает с effective device limit пользователя, сохраняет `модель / ОС / версия ОС` и не выдаёт рабочий bundle следующему уникальному устройству сверх этого лимита;
- install-flow на client page теперь держит platform-specific official sources прямо в summary payload: `Android`, `iOS`, `Windows`, `macOS`, `Linux`, `Apple TV`, `Android TV` получают короткое описание установки и список живых ссылок (`Google Play / App Store / GitHub releases / APK`) без редиректа в общий `/manual`;
- summary payload tokenized client page теперь должен отдавать два device-среза одновременно: legacy-совместимый `bound_devices*` для слотов единой ссылки и канонический `account_devices*` для общего лимита аккаунта, чтобы `client.amonoraconnect.com` и операторская таблица пользователей показывали одно и то же количество занятых устройств;
- текущий human-readable server list для этого surface синхронизирован между client page и Happ feed и сейчас intentionally narrowed до трёх primary routes без user-facing reserve-entries: `#1 Германия` = текущий Germany `VLESS + Reality + TCP`, `#1 Дания` = текущий Denmark `VLESS + Reality + XHTTP`, `#1 Эстония` = текущий Estonia `VLESS + Reality + TCP + xtls-rprx-vision`;
- `v1` не заменяет текущий device-flow в `@amonora_bot`, а живёт параллельно ему;
- этот surface не должен становиться отдельной веб-оплатой, support-кабинетом или публичной общей витриной.

## Telegram-канал

Telegram-канал [`@amonora_new`](https://t.me/amonora_new) является частью публичного присутствия бренда.

На текущем этапе он нужен для:

- публикации обновлений;
- анонсов;
- новостей проекта;
- формирования доверительного публичного контура вокруг продукта.

Текущее уточнение:

- часть постов канала теперь может публиковаться с tracked CTA-кнопкой в `@amonora_bot`, чтобы команда видела переходы и конверсии канала без смешивания этого потока с referral/campaign attribution.

## Landing bridge-access

Публичный сайт `amonoraconnect.com` теперь поддерживает ограниченный bridge-access сценарий.

Он нужен для случаев, когда пользователь не может сразу открыть Telegram, но может открыть сайт.

Текущие правила этого surface:

- сайт может выдать бесплатный временный ключ подключения на `24 часа`;
- этот ключ нужен только как мост до `@amonora_bot`, а не как отдельный тарифный продукт;
- после выдачи ключа landing показывает кнопку перехода в `@amonora_bot` и кнопку на инструкцию `/manual`;
- bridge-flow должен быть rate-limited и не должен разрастаться в отдельный полноценный web-cabinet;
- synthetic bridge users и bridge devices не должны искажать обычные пользовательские агрегаты панели управления.

## Отзывы

Вопрос с отзывами пока не закреплён как единый канонический сценарий.

По текущему состоянию проекта это открытая продуктовая зона, которая требует отдельного решения.

Возможные варианты:

- отдельный сценарий в `support_bot`;
- форма на сайте;
- отдельный Telegram-формат;
- публичная страница отзывов.

## Текущее правило

Пока канал отзывов не утверждён, не стоит считать ни одну из текущих поверхностей официальным и единственным местом для отзывов.

При этом для формальной публичной обратной связи действует отдельный email:

- `amonoraconnect@yandex.ru`

Отдельное правило для скрытой инструкции:

- `/manual` не должен появляться как постоянный пункт основной публичной навигации;
- при этом ссылка на `/manual` может показываться как контекстный следующий шаг после выдачи bridge-ключа или при прямой передаче пользователю;
- эта страница остаётся частью onboarding/support handoff, а не самостоятельным публичным разделом продукта.

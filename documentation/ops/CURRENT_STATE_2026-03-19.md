## 1) Краткое резюме состояния

Проект уже вышел из стадии черновика. Это не тестовый стенд, а ранний боевой сервис:

- есть реальные пользователи
- есть минимум один подтвержденный платёж
- есть разделение `core/backend` и `VPN nodes`
- есть `Telegram bot`, `support-bot`, `dashboard v2`, `PostgreSQL`, `3x-ui + Xray`
- есть мониторинг и operational слой

При этом система пока не готова к безопасному масштабированию без этапа стабилизации. Главные риски — не нехватка функционала, а эксплуатационная хрупкость, VPN-совместимость на краях и грязное состояние репозитория.

## 3) Текущий VPN-слой

### 3.1 Что реально в проде

#### Германия
- `VLESS + Reality`
- `443/tcp`
- `network=tcp`
- `security=reality`
- SNI: `www.microsoft.com`
- набор нескольких `shortId`

#### Германия fallback
- `Trojan + TLS`
- `8443/tcp`
- `network=tcp`
- `security=tls`

#### Эстония
- `VLESS + Reality`
- `443/tcp`
- `network=tcp`
- `security=reality`
- SNI: `www.microsoft.com`
- сейчас тот же набор `shortId`

#### Эстония fallback
- `Trojan + TLS`
- `8443/tcp`
- `network=tcp`
- `security=tls`

#### Legacy хвост
- на Эстонии ещё висел legacy UDP inbound на `51820`

### 3.2 Что уже пробовали и что важно помнить

#### XHTTP
Пробовали на Эстонии, но получили:
- `EOF`
- `TLS handshake`

Вывод:
- на твоём реальном клиентском контуре XHTTP пока **не готов для продового использования**

#### PQ-режим / mlkem
Была практическая проблема, когда `3x-ui` начал выдавать ссылки с новым PQ-режимом (`mlkem...`), а клиенты это не переваривали.

Вывод:
- обновления `3x-ui` / Xray могут **тихо ломать рабочие конфиги**
- режимы надо менять только после точечного теста

## 5) Главные риски

### 5.1 Single Point of Failure
Backend один.

Если падает core:
- падает бот
- падает support
- падает dashboard
- падают платежные и control-plane сценарии

Даже если VPN-ноды физически ещё живы, сервис как продукт становится поломанным.

### 5.2 Один IP на ноду
- Германия = один IP
- Эстония = один IP

Любая сильная репутационная просадка сразу бьёт по целому региону.

### 5.3 Слишком одинаковая маскировка между нодами
Сейчас между Germany и Estonia слишком много сходства:
- одинаковый SNI
- одинаковый набор `shortId`
- одинаковый transport (`tcp`)

Это снижает разнообразие маскировки.

### 5.4 Слабая Эстония
Эстония полезна как резерв, но:
- `1 vCPU`
- `~1 GB RAM`
- `100 Mbps`

Это не нода, на которую можно бездумно посадить заметную долю роста.

### 5.5 Грязный git / процессный риск
Это один из самых практических рисков.

Пока дерево грязное:
- опасно делать большие изменения
- сложно откатываться
- сложно понять, что реально прод, а что недоделка
- легко случайно сломать рабочий контур

## 6) Что сейчас важнее всего

Главный приоритет сейчас:

> **не внедрять новые фичи, а стабилизировать рабочий прод без ломки пользователей**

Это означает:
- не трогать резко рабочие транспорты
- не включать заново XHTTP
- не обновлять бездумно `3x-ui` / Xray
- не менять массово текущие пользовательские конфиги
- сначала сделать rollback safety, инвентаризацию, regression и дисциплину изменений

## 8) Следующий шаг №1 — схема SNI + Reality под текущий проект

Ниже — **консервативная схема**, которая рассчитана не на «идеальную теорию», а на реальный ранний прод, где уже есть пользователи.

### 8.1 Главный принцип

Сейчас нельзя делать резкую массовую замену уже выданных конфигов.

Поэтому стратегия такая:

- **старые рабочие inbound не ломаем**
- **для новых выдач готовим более раздельный профиль**
- **меняем только после точечного теста на своих устройствах**

### 8.2 Что оставить как есть прямо сейчас

Оставить без немедленной перестройки:

- Germany `VLESS + Reality + TCP`
- Estonia `VLESS + Reality + TCP`
- Germany `Trojan + TLS`
- Estonia `Trojan + TLS`
- текущие активные пользовательские конфиги

### 8.3 Что улучшить без ломки старых клиентов

#### Шаг 1. Развести shortId между Germany и Estonia
Сейчас одинаковый набор `shortId` на двух нодах — это плохая идея.

Правильнее:
- Germany: оставить текущий набор как legacy-совместимый
- Estonia: сделать **новый отдельный набор shortId**

Как внедрять:
1. не менять старый активный inbound для существующих клиентов
2. создать новый inbound или новый профиль на Эстонии
3. проверить на своих устройствах
4. выдавать только новым клиентам / новым устройствам
5. после наблюдения решать, переводить ли часть потока дальше

#### Шаг 2. Развести SNI между нодами
Сейчас обе ноды используют `www.microsoft.com`.

Консервативный путь:
- Germany пока оставить на текущем `www.microsoft.com`
- Estonia для нового профиля перевести на **другой нормальный крупный SNI**, не трогая старый inbound

Смысл:
- не делать обе ноды полностью одинаковыми
- не рисковать одномоментно рабочим продом

#### Шаг 3. Не гнаться за экзотикой
Сейчас не надо:
- возвращать `XHTTP`
- включать новый PQ-режим
- прыгать на `HTTP/3`
- массово менять client fingerprints

Текущая задача не «сделать самый навороченный стек», а **сделать более разнообразный и предсказуемый стек**.

### 8.4 Как внедрять изменения по Reality безопасно

Правило внедрения:

1. backup параметров старого inbound  
2. создание нового профиля отдельно  
3. тест на своих устройствах  
4. тест с одним неключевым новым устройством  
5. наблюдение 24–48 часов  
6. только потом постепенное использование для новых выдач

### 8.5 Практический шаблон принятия решений

#### Germany
Роль:
- основная стабильная нода

Тактика:
- **минимум изменений**
- сохранить как главный стабильный маршрут
- не делать на ней рискованных экспериментов

#### Estonia
Роль:
- резерв / экспериментально-усиленный маршрут

Тактика:
- именно здесь можно аккуратно пробовать раздельный профиль
- но только без возврата к XHTTP прямо сейчас

### 8.6 Что считать успехом этого шага

Успех — это не «всё перевели на новый стиль».  
Успех — это:

- Germany и Estonia стали менее одинаковыми
- старые пользователи не пострадали
- новые выдачи идут через более раздельные профили
- ты получил больше контроля без резкой миграции

## 9) Следующий шаг №2 — как безопасно менять конфиги без падения пользователей

Это должен быть твой постоянный operational регламент.

### 9.1 Базовое правило

Любое изменение проходит цикл:

**backup → change → smoke test → observe → only then next step**

Не делать:
- несколько рискованных изменений подряд
- изменение конфига и одновременно обновление панели
- изменение панели и одновременно изменение onboarding flow

### 9.2 Перед любым изменением — обязательный pre-change checklist

#### Backup / фиксация
- [ ] Есть свежий дамп Postgres
- [ ] Сохранены `.env` / env-файлы
- [ ] Сохранены `nginx` конфиги
- [ ] Сохранены `systemd` unit-файлы
- [ ] Сохранены текущие параметры inbound-ов
- [ ] Зафиксирован текущий `git status`
- [ ] Понятно, как откатить именно это изменение

#### Риск-оценка
- [ ] Изменение затронет существующих пользователей или только новых?
- [ ] Меняется transport / security / SNI / shortId / panel behavior?
- [ ] Есть ли зависимость от клиента (Happ / desktop / mobile)?
- [ ] Это точно не эксперимент на живом основном канале?

#### План проверки
- [ ] Есть минимум один собственный тестовый клиент
- [ ] Есть тест на Android / desktop, если релевантно
- [ ] Есть короткий smoke test после деплоя
- [ ] Есть критерий успеха и критерий отката

### 9.3 Безопасная последовательность изменения VPN-конфига

#### Сценарий: добавить новый профиль / изменить параметры для новых выдач
1. сохранить старый inbound
2. создать **новый** inbound / новый профиль, а не ломать старый
3. проверить, что старый продолжает обслуживать текущих клиентов
4. выдать новый профиль только себе
5. провести тест:
   - импорт
   - коннект
   - открытие 2–3 реальных сервисов
   - повторное подключение
   - QR / deep-link
6. выдать одному безопасному тестовому сценарию
7. понаблюдать
8. только после этого использовать для новых реальных выдач

#### Никогда не делать в бою сразу
- менять существующий inbound для всех клиентов
- удалять старый inbound до проверки нового
- включать новый transport без полного теста клиента
- менять сразу и параметры Reality, и panel version, и onboarding

### 9.4 Безопасная последовательность изменения dashboard / backend

#### Перед изменением
- [ ] понятен affected route
- [ ] есть конкретный список ручных проверок
- [ ] известен способ быстрого отката build / сервиса

#### После выката
Проверить минимум:
- [ ] login
- [ ] request code
- [ ] verify
- [ ] overview
- [ ] users
- [ ] extend tariff
- [ ] payments
- [ ] support
- [ ] notifications

Если хоть один ключевой денежный или доступный сценарий сломан — откатывать быстро, а не «допиливать на живом».

### 9.5 Регламент по 3x-ui / Xray

Пока проект в этой стадии:

- не обновлять `3x-ui` без причины
- не обновлять Xray без понятного повода
- любое обновление сначала считать потенциально опасным
- после обновления проверять, не изменился ли формат выдаваемых ссылок
- особенно внимательно смотреть на:
  - `decryption`
  - `Reality`
  - новые security-поля
  - client compatibility

### 9.6 Практическое правило одного изменения

Лучший режим для тебя сейчас:

> **одно логическое изменение за раз**

Примеры:

Хорошо:
- поменять только shortId-профиль для нового Estonia inbound
- выкатить только фикс login-flow
- удалить только один legacy path после проверки

Плохо:
- обновить панель
- поменять конфиг Reality
- поменять onboarding
- перевыпустить dashboard bundle
- и всё это за один вечер

### 9.7 Что делать, если что-то пошло не так

#### Признаки инцидента
- новый конфиг не импортируется
- connect есть, трафика нет
- часть сервисов не открывается
- начались жалобы после одной и той же правки
- dashboard начал давать старый маршрут / stale behavior

#### Действия
1. остановить следующие изменения
2. зафиксировать время и последнюю правку
3. определить, затронуты старые клиенты или только новые
4. откатить именно последнее изменение
5. перепроверить ключевой пользовательский сценарий
6. только потом анализировать причину

### 9.8 Мини-шаблон change record

Используй такой формат для каждого изменения:

## 11) Напоминания

### Главные правила на этой стадии

- пользователи важнее экспериментов
- стабильность важнее новых фич
- откат важнее «быстро дожать фикс на проде»
- один безопасный шаг лучше пяти красивых, но рискованных
- то, что работает, не трогать без причины

### Что пока не трогать
- не включать снова `XHTTP`
- не включать `HTTP/3` в прод для реальных пользователей
- не включать без теста новые PQ-режимы

- не ломать существующие inbound-ы массово

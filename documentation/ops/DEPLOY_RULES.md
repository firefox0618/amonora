# AMONORA — CHANGELOG & DEPLOY RULES
Дата: 19 марта 2026

---

# ЦЕЛЬ

Этот документ вводит простую дисциплину:

> Любое изменение должно быть контролируемым, проверяемым и откатываемым.

Без этого проект не масштабируется и ломается на росте.

---

# 1. ОСНОВНОЕ ПРАВИЛО

> Нет деплоя без записи в changelog.

Если изменение не записано — его нельзя делать.

---

# 2. ФОРМАТ CHANGELOG

Каждое изменение записывается так:

## Пример

Дата: 2026-03-19  
Сервер: Germany  
Компонент: VPN / Xray  
Изменение: создан новый inbound vless-reality-tcp-de-new  
Причина: разделение SNI и shortId  
Риск: низкий  
Проверка:
- подключение работает
- старый inbound не сломан  

Откат:
- удалить новый inbound  
- вернуть старую выдачу  

Статус: OK / FAIL

---

# 3. КАТЕГОРИИ ИЗМЕНЕНИЙ

Каждое изменение должно быть классифицировано:

## LOW RISK
- добавление нового inbound
- UI правки
- логика без влияния на прод

## MEDIUM RISK
- изменение выдачи конфигов
- изменение bot flow
- изменения nginx

## HIGH RISK
- изменение текущих inbound
- обновление Xray / 3x-ui
- изменение портов
- массовая миграция пользователей

---

# 4. ПРАВИЛО ДЕПЛОЯ

Каждое изменение проходит:

## 1. Подготовка
- [ ] есть backup
- [ ] есть план отката
- [ ] записано в changelog

## 2. Применение
- [ ] одно изменение за раз
- [ ] без параллельных правок

## 3. Проверка
- [ ] проверен основной сценарий
- [ ] проверен fallback
- [ ] проверен bot flow

## 4. Фиксация
- [ ] обновить changelog статус
- [ ] записать результат

---

# 5. ЗАПРЕТЫ

❌ делать несколько изменений подряд  
❌ менять и VPN, и backend одновременно  
❌ деплоить без теста  
❌ менять прод “на глаз”  
❌ работать без backup  

---

# 6. ПРАВИЛО ОДНОГО ИЗМЕНЕНИЯ

> В один момент времени меняется только одна вещь.

Пример:

❌ Плохо:
- поменял SNI
- обновил Xray
- изменил бот

✔ Хорошо:
- поменял SNI → проверил → зафиксировал  
- потом следующий шаг  

---

# 7. ВРЕМЯ ДЕПЛОЯ

Рекомендуется:

- ночью или в низкую нагрузку
- не во время активных пользователей
- не во время платежей

---

# 8. КОНТРОЛЬ ПОСЛЕ ДЕПЛОЯ

После любого изменения проверить:

## VPN
- [ ] подключается
- [ ] нет ошибок handshake
- [ ] скорость нормальная

## Бот
- [ ] выдает конфиг
- [ ] создаёт устройство
- [ ] не падает

## Dashboard
- [ ] логин работает
- [ ] действия работают
- [ ] нет 500 ошибок

---

# 9. ЛОГ ИНЦИДЕНТОВ

Если что-то сломалось:

Записать:

- что сломалось
- когда
- после какого изменения
- как исправили
- как избежать в будущем

---

# 10. МИНИМАЛЬНАЯ ДИСЦИПЛИНА

Даже если ты один разработчик:

- веди changelog
- делай backup
- не торопись
- проверяй руками

---

# 11. ФОРМУЛА СТАБИЛЬНОСТИ

> меньше изменений = меньше багов  
> контроль = стабильность  
> стабильность = доверие пользователей

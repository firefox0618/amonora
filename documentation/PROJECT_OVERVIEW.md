# PROJECT OVERVIEW

## Что такое Amonora

`Amonora` — это экосистема цифровых сервисов.
Текущий основной продукт — Telegram-first VPN-сервис, где клиентский путь построен вокруг `@amonora_bot`: пользователь получает trial, подключает устройство, выбирает страну и протокол, управляет подпиской и поддержкой — без отдельного «классического» личного кабинета.

## Пользователи

| Роль | Что делает |
|------|-----------|
| **Клиент** | Получает trial, подключает устройства, выбирает страну/протокол, продлевает подписку, обращается в поддержку |
| **Поддержка** | Принимает обращения, ведёт тикеты, отвечает клиентам через `@amonora_support_bot` |
| **Администраторы** | Управляют системой через `dashboard` и `@amonora_control_bot`: пользователи, платежи, серверы, настройки |

## Контуры системы

| Контур | Назначение |
|--------|-----------|
| `@amonora_bot` | Главный клиентский бот |
| `@amonora_support_bot` | Бот поддержки |
| `@amonora_control_bot` | Системные уведомления, review платежей, коды входа в админку |
| `@test_amonora_bot` | Тестовый бот для мобильных профилей (admin-only) |
| `amonoraconnect.com` | Публичный сайт-витрина |
| `client_ui` | Исходники tokenized client page (React/TypeScript) |
| `dashboard` + `dashboard/ui` | Панель управления (backend + Next.js frontend) |
| VPN-ноды | Отдельные серверы: Германия (3x-ui), Дания (Xray core), Эстония (reserve) |

## Основной сценарий

1. Пользователь приходит через сайт или `@amonora_bot`
2. Получает trial или оформляет подписку
3. Выбирает устройство, страну, протокол
4. Backend проверяет пользователя и сохраняет данные в PostgreSQL
5. Пользователь получает конфиг, QR или статус в Telegram
6. При проблемах — переходит в `@amonora_support_bot`

## Где ядро и данные

- **Ядро**: `backend` (модели, схема БД) + значимая логика в `bot`, `dashboard`
- **Данные**: PostgreSQL — единственный source of truth
- **Интерфейсы**: `bot`, `support_bot`, `control_bot`, `landing`, `dashboard/ui`
- `dashboard/ui` и `landing` не являются источником истины

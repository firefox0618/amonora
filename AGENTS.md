# AGENTS.md

## Назначение

Этот файл задаёт правила для AI-агентов и разработчиков, работающих в репозитории `amonora`.

Репозиторий совмещает несколько контуров:

- `bot/` — основной клиентский Telegram-бот.
- `support_bot/` — бот поддержки.
- `control_bot/` — внутренний control bot.
- `test_bot/` — тестовый Telegram-бот.
- `backend/` — backend и общая серверная логика.
- `dashboard/` — backend/admin surface и отдельный UI.
- `dashboard/ui/` — Next.js UI для панели управления.
- `client_ui/` — клиентский web UI на Vite/React.
- `landing/` — публичный сайт и статические клиентские поверхности.
- `ops/` — эксплуатационные, backup, nginx, systemd и service-скрипты.
- `tests/` — тестовая поверхность.
- `docs/` — основная база знаний проекта.
- `.obsidian/` — конфигурация vault, подтверждающая что документация используется через Obsidian.

## Главные правила работы с кодом

- Редактируй существующий код, а не добавляй новый поверх старого.
- Не дублируй функции, компоненты, классы, хуки, типы и стили.
- Если нужно изменить поведение, найди существующую реализацию и измени её.
- Перед созданием нового файла или функции проверь, нет ли уже похожей реализации.
- Удаляй устаревший код, если он заменён новой реализацией.
- Не оставляй две версии одной и той же логики.

## Как вносить изменения

- Делай минимальные точечные изменения.
- Сохраняй текущую архитектуру проекта, если явно не попросили рефакторинг.
- Не переписывай весь файл без необходимости.
- Не меняй публичные API без причины.
- Не ломай существующие импорты и экспорты.
- После изменений проверь, что не осталось неиспользуемого кода.
- Если задача локальна для одного контура, не трогай соседние модули без необходимости.
- Если меняешь shared-логику, проверь все потребители, а не только один вход.

## Границы по контурам

- В `bot/` сначала ищи логику в `bot/router.py`, а потом уже в legacy-обработчиках и вспомогательных модулях.
- В `support_bot/`, `control_bot/` и `test_bot/` не копируй код из `bot/`; выноси общее поведение только если оно действительно разделяется между контурами.
- В `backend/` и `dashboard/` сохраняй текущие контракты API, модели данных и бизнес-правила.
- В `dashboard/ui/` сначала переиспользуй существующие UI-паттерны, утилиты и компоненты, прежде чем создавать новые.
- В `client_ui/` и `landing/` не дублируй визуальную логику между клиентской страницей подписки и публичным сайтом без необходимости.
- В `ops/` любые изменения должны быть особенно точными: не меняй пути, unit names, nginx/server-конфиги и backup-скрипты без прямой причины.

## Дизайн и UI

- Не добавляй новые стили поверх старых, если можно изменить существующие.
- Переиспользуй существующие компоненты.
- Сохраняй единый визуальный стиль проекта.
- Не создавай дублирующиеся CSS-классы, utility-слои или компоненты.
- Для улучшения дизайна сначала проверь текущие компоненты, layout и стили.
- Для `dashboard/ui/` учитывай существующий стек: `Next.js`, `React`, `TypeScript`.
- Для `client_ui/` учитывай существующий стек: `Vite`, `React`, `Mantine`.
- Если правишь шаблоны или статику в `landing/`, сохраняй совместимость с текущими маршрутами, шаблонами и статическими asset-путями.

## Работа с документацией и Obsidian

- Считай `docs/` канонической проектной базой знаний.
- Учитывай, что репозиторий открыт как Obsidian vault: не ломай `.obsidian/`, структуру каталогов и навигационные заметки без явного запроса.
- При изменении `.md` файлов сохраняй существующие относительные ссылки, заголовки и структуру разделов.
- Не переименовывай документы и папки в `docs/`, если это не требуется задачей: это может сломать Obsidian-ссылки и ручную навигацию.
- Если добавляешь новую заметку в `docs/`, помещай её в подходящий тематический раздел, а не в корень без причины.
- Обновляй `docs/index.md`, `docs/STRUCTURE.md` или связанные обзорные заметки, если изменение реально влияет на карту документации.
- Не засоряй `docs/` временными черновиками, служебными дампами и autogenerated-файлами.
- Изображения и медиа для документации складывай только в предусмотренные каталоги внутри `docs/images/`, если это действительно нужно.

## Правила для Obsidian-заметок

- Рассматривай каждый Markdown-файл в `docs/` как заметку, которая должна быть удобна и в git, и в Obsidian.
- Не меняй названия файлов, если это не требуется задачей: в Obsidian имя файла часто является частью навигации и ручных ссылок.
- Не перемещай заметки между разделами без явной причины и без обновления связанных обзорных файлов.
- Сохраняй читаемую иерархию заголовков: не пропускай уровни без необходимости.
- Не превращай заметки в сырые технические дампы; документация должна оставаться читаемой для команды.
- При добавлении новой заметки используй понятное русскоязычное имя файла в стиле уже существующей документации.
- Поддерживай единый стиль ссылок внутри раздела: не смешивай хаотично разные форматы ссылок, если в текущем файле уже есть устоявшийся способ.
- Если правка затрагивает обзорный документ, проверь, не появились ли битые ссылки на соседние заметки.
- Не удаляй служебные обзорные заметки, такие как `docs/index.md` и `docs/STRUCTURE.md`, без явного запроса.
- Не меняй `.obsidian/app.json`, `.obsidian/appearance.json`, `.obsidian/core-plugins.json` и `.obsidian/workspace.json`, если задача не связана именно с настройкой Obsidian.
- Не складывай вложения рядом с заметками россыпью, если для них уже есть выделенные каталоги вроде `docs/images/`.
- Если добавляешь новую папку в `docs/`, делай это только когда существующие разделы действительно не подходят.

## Работа с файлами

- Используй filesystem MCP для чтения нужных файлов вместо угадывания.
- Используй git MCP для анализа diff до и после изменений.
- Если задача касается библиотеки или фреймворка, используй context7 для актуальной документации.
- Перед созданием нового файла проверь, нет ли уже подходящего файла в текущем контуре.
- Не создавай новые каталоги, если существующая структура уже покрывает задачу.
- Не трогай `.env` и другие секреты без явного запроса.
- Не коммить и не описывай как норму артефакты из `.venv/`, `__pycache__/`, `.pytest_cache/` и локальных runtime-данных.

## Перед изменением кода

Сначала:
1. Найди существующую реализацию.
2. Определи, что именно нужно изменить.
3. Измени код на месте.
4. Удали старую или лишнюю реализацию.
5. Проверь diff.

## Тесты и проверка

- После правок запускай минимально достаточную проверку, которая соответствует затронутому контуру.
- Для Python-кода базовая проверка — релевантные тесты из `tests/` через `pytest`.
- Для `dashboard/ui/` используй как минимум `npm run typecheck`, `npm run lint` или другую релевантную проверку, если затронут TypeScript/UI-код.
- Для `client_ui/` как минимум проверяй сборку или другой релевантный сценарий для затронутых изменений.
- Если меняешь `ops/`, `nginx`, `systemd`, backup или deployment-related код, отдельно укажи, что нужно проверить вручную.
- Если тесты не запускались, явно сообщи об этом.

## Запрещено

- Не добавлять "новую версию" функции рядом со старой.
- Не оставлять старый код закомментированным.
- Не создавать `ComponentNew`, `ComponentV2`, `newFunction` и похожие сущности, если можно изменить существующее.
- Не плодить файлы без необходимости.
- Не делать большие переписывания без явного запроса.
- Не переносить логику между контурами только ради косметики.
- Не вносить массовые переименования, если задача точечная.
- Не менять структуру `docs/` и `.obsidian/` без явной причины.

## После изменений

- Покажи краткий список изменённых файлов.
- Объясни, что было изменено.
- Укажи, если что-то нужно проверить вручную.
- Если запускались тесты или проверки, перечисли их кратко.
- Если есть ограничения, риски или места без верификации, назови их прямо.

# Amonora TZ V2

## Техническое задание v5 — Amonora Control

## Статус документа

- язык документа: русский;
- документ актуализирует и расширяет `amonora_control_tz_v4.md`;
- этот файл является текущей продуктовой спецификацией для новой панели `Amonora Control`;
- формулировка `старая панель удалена, создана новая чистая` относится к активному операторскому UI;
- backend/API seam `dashboard` остаётся рабочим источником истины и не должен ломаться ради визуального редизайна.

## 0. Главная цель

Панель управления должна быть операционным центром `Amonora Connect`.

Ключевой принцип:

> Деньги -> доступ -> работает  
> Если не работает -> одна понятная кнопка должна либо восстановить доступ, либо явно эскалировать проблему.

Цель панели:

- быстро подтверждать ручные платежи;
- быстро восстанавливать доступ пользователя;
- видеть состояние нод и сервисов;
- работать с support без прыжков между разными интерфейсами;
- поддерживать несколько ролей администраторов;
- давать премиальный, спокойный, современный SaaS-интерфейс.

## 1. Контекст и ограничения

- продукт Telegram-first, основной пользовательский контур живёт в `@amonora_bot`;
- поддержка ведётся через `@amonora_support_bot` и панель;
- внутренний review и системные сигналы идут через `@amonora_control_bot`;
- активных пользователей нельзя ломать;
- доступ должен пропадать серверно при окончании подписки;
- любой визуальный редизайн не должен разрушать текущий Python/backend-контур.

### 3.1 Frontend

- `Next.js` (`App Router`);
- `TypeScript`;

- `1 ключ = 1 устройство` строго;
- ключ не должен работать как мульти-доступ;
- при перевыпуске старый ключ перестаёт работать сразу;
- при удалении устройства ключ удаляется безвозвратно, затем пользователь создаёт новое устройство заново;
- лимит на перевыпуск по суткам не требуется.

### 8.7 Поведение при окончании подписки

- доступ пропадает;
- серверная сторона обязана учитывать окончание доступа;
- устройство может оставаться в истории, но рабочий доступ должен быть отключён.

## 9. Sync / Repair

### 9.1 Смысл кнопки `Синхронизировать`

Кнопка нужна для реального восстановления доступа, если:

- тариф активен;
- доступ оплачен;
- устройство уже было рабочим;
- потом доступ перестал работать.

### 9.2 Что делает `Синхронизировать`

Базовый сценарий:

- проверяет статус пользователя;
- проверяет доступ и тариф;
- проверяет лимит и наличие устройств;
- сверяет данные устройств с БД;
- проверяет применение доступа на ноде;
- при необходимости пересоздаёт ключ;
- перепривязывает к ноде;
- заново применяет доступ на стороне VPN;
- при необходимости пересобирает конфиг;
- пишет результат в историю;
- если нужно, отдаёт пользователю обновлённый рабочий конфиг.

### 9.3 Что считать `sync_error`

- доступ в БД есть, но на ноде не применился;
- ключ создан, но не активен;
- нода не ответила;
- конфиг битый;
- платёж подтверждён, но доступ не выдался;
- устройство есть, но не совпадает с реальным состоянием на ноде;
- любой другой backend-trigger repair failure.

### 9.4 Кнопка `Глубокий ремонт`

Это более сильный сценарий, чем обычный `sync`.

Должен уметь:

- полностью пересобрать access state;
- перепривязать устройство к корректной ноде;
- заново выпустить конфиг;
- повторно применить доступ;
- выполнить дополнительные backend repair-операции по триггеру причины ошибки.

### 9.5 Если repair не помог

Система должна:

- выставить статус `repair_needed`;
- зафиксировать причину;
- записать событие в audit / history;
- отправить алерт администратору;
- оставить кейс видимым в `Требует внимания`.

### 9.6 История repair

Нужно хранить:

- количество попыток починки;
- дату последней успешной синхронизации;
- дату последней неуспешной синхронизации;
- последние 5 записей по repair-истории.

## 10. Платежи и финансы

### 10.1 Текущие способы оплаты

- `Telegram Stars`;
- ручной `СБП`;
- ручная крипта.

### 10.2 Основной ручной СБП flow

Шаги:

1. пользователь нажимает тариф в боте;
2. пишет в поддержку;
3. получает реквизиты;
4. оплачивает;
5. возвращается в `@amonora_bot`;
6. нажимает `Я оплатил`;
7. в `@amonora_control_bot` приходит запрос на review;
8. админ подтверждает или отклоняет платёж.

### 10.3 Что происходит после подтверждения

Автоматически:

- выдать доступ или продлить текущий;
- довести пользователя до рабочего состояния;
- сделать `sync` после оплаты;
- записать событие в историю;
- уведомить пользователя;
- при необходимости дать сигнал в support-контур.

### 10.4 Статусы платежей

Основные:

- `awaiting_user_payment`;
- `awaiting_admin_review`;
- `confirmed`;
- `rejected`;
- `expired`;
- `disputed`;
- `error`.

Дополнительные статусы для будущих auto-flow:

- `down`;
- `maintenance`.

### 12.3 Что считать проблемной нодой

- нет heartbeat;
- высокий CPU / RAM / Disk;
- сервис `down`;
- не проходит health check;
- отвалился sync worker;
- любой иной backend health signal.

### 12.4 KPI по серверам

- активные ноды;
- ноды с проблемами;
- всего устройств;
- активные устройства;
- средняя нагрузка.

### 12.5 Карточка ноды

- регион;

- статус;
- CPU;
- RAM;
- Disk;
- throughput;

### 12.6 Detail panel ноды

- метрики;
- сервисы;
- список пользователей / устройств на ноде;
- ошибки;

- быстрые действия.

### 12.7 Действия по ноде

- `restart service`;
- `health check`;
- `maintenance mode`;
- `resync` по ноде;
- перенос пользователей / устройств на другую ноду;
- mark maintenance.

- интерфейс должен оставаться быстрым даже при росте таблиц пользователей и платежей.

## 19. Бизнес-правила

- доступ появляется только после подтверждения оплаты, если это ручной сценарий;
- если доступ есть, он должен реально работать;
- если не работает, `sync` обязан быть первой операционной кнопкой;
- trial не должен выдаваться повторно;
- продление добавляется к текущему сроку или отражается через баланс, если так устроен backend flow;
- история по ключевым сущностям хранит минимум последние 5 записей там, где это применимо;
- после подтверждения оплаты пользователь должен получить сообщение о том, что доступ активирован или продлён.

## 20. Критерии готовности ядра

Ядро панели считается готовым, если:

- ручной СБП flow полностью проходит через `bot -> control bot -> panel/backend`;
- админы видят и подтверждают платежи;
- пользовательский доступ реально восстанавливается через `sync`;
- users / payments / support / servers связаны между собой контекстно;
- audit и alerts работают;
- панель имеет светлую и тёмную тему;
- ключевые KPI и health-срезы обновляются без ручного рефреша.

## 22. Краткий итог

Новая панель должна быть:

- красивой;
- быстрой;
- понятной;
- технологичной;
- безопасной для активных пользователей;
- удобной для owner / tech admin / manager;
- построенной вокруг реальной операционной работы.

> Деньги -> доступ -> работает  
> Если не работает -> одна кнопка либо чинит, либо эскалирует проблему без потери контекста.
